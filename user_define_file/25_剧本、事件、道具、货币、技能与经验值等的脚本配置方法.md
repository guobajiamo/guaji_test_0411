# 25_剧本、事件、道具、货币、技能与经验值等的脚本配置方法

## 1. 先说最重要的结论

当前项目仍然是：

- `.tres` 负责“这次进入哪套剧本”
- 剧本 YAML 负责“这套剧本里到底有什么内容”
- 新 UI 的 `gameplay_layout.yaml` 负责“这些内容在游戏内容界面里摆在哪里”

所以你后面继续扩写内容时，最常见的工作仍然会集中在：

1. 新建或复制一个剧本 `.tres`
2. 给它准备一套 `Configs/Scenarios/你的剧本名/` 目录
3. 在 `Items / Skills / Events / Factions / Zones / Localization / UI` 里配置内容

当前最推荐的工作方式仍然是：

- 系统脚本尽量少改
- 内容优先改 YAML
- 新 UI 的区域层级、子场景、事件归属优先改 `UI/gameplay_layout.yaml`

## 2. 当前真正应该改的剧本配置目录在哪里

### 2.1 主线剧本

主线剧本资源文件：

- `Resources/Scenarios/main_story_scenario.tres`

主线剧本当前使用的配置根目录：

- `Configs/Scenarios/MainStory/`

### 2.2 测试剧本

测试剧本资源文件：

- `Resources/Scenarios/test_game_scenario.tres`

测试剧本当前使用的配置根目录：

- `Configs/Scenarios/TestGame/`

## 3. 新建一个新剧本，建议按这个步骤做

### 第一步：复制一个现有剧本资源

最稳妥的做法，是先复制：

- `Resources/Scenarios/main_story_scenario.tres`

例如你想做一个新剧本 `forest_story`，可以新建：

- `Resources/Scenarios/forest_story_scenario.tres`

### 第二步：修改剧本资源里的基础字段

你需要重点改这些字段：

- `ScenarioId`
- `DisplayName`
- `Description`
- `ShowInNewGameMenu`
- `IsDefaultStoryScenario`
- `IsTestScenario`
- `EnableSystemTab`

示例：

```text
ScenarioId = "forest_story"
DisplayName = "森林支线剧本"
Description = "用于测试森林区域扩写的版本"
ShowInNewGameMenu = true
IsDefaultStoryScenario = false
IsTestScenario = false
EnableSystemTab = true
```

### 第三步：给新剧本准备独立 YAML 目录

建议新建：

```text
Configs/Scenarios/ForestStory/
  Items/
    categories.yaml
    items.yaml
  Skills/
    skills.yaml
  Events/
    oneshot_events.yaml
    click_events.yaml
    idle_events.yaml
  Factions/
    factions.yaml
  Zones/
    zones.yaml
  Localization/
    locale_zh.yaml
  UI/
    info_panel.yaml
    gameplay_layout.yaml
```

### 第四步：把 `.tres` 指向你的 YAML

剧本资源里需要重点检查这些路径：

- `CategoriesConfigPath`
- `ItemsConfigPath`
- `SkillsConfigPath`
- `OneshotEventsConfigPath`
- `ClickEventsConfigPath`
- `IdleEventsConfigPath`
- `FactionsConfigPath`
- `ZonesConfigPath`
- `LocalizationPath`
- `InfoPanelConfigPath`

注意：

- 当前没有单独的 `GameplayLayoutPath`
- 程序会根据 `InfoPanelConfigPath` 所在目录
- 自动读取同目录下的 `gameplay_layout.yaml`

也就是说，只要你把：

- `InfoPanelConfigPath = "res://Configs/Scenarios/ForestStory/UI/info_panel.yaml"`

写对，那么：

- `res://Configs/Scenarios/ForestStory/UI/gameplay_layout.yaml`

就会被自动读取。

## 4. 如何配置物品分类 `categories.yaml`

物品分类决定的是“物品在树状结构里属于哪一类”。

示例：

```yaml
categories:
  - id: "tool"
    parent_id: ""
    name_key: "工具"
    description_key: "用于采集与互动的工具"
    definition_order: 100
    is_visible: true
```

字段说明：

- `id`：分类唯一 ID，不能重复
- `parent_id`：父分类 ID，没有父级就留空
- `name_key`：显示名称
- `description_key`：分类描述
- `definition_order`：排序
- `is_visible`：是否默认显示

## 5. 如何配置物品 `items.yaml`

示例：

```yaml
items:
  - id: "tool_stone_axe"
    parent_id: "tool"
    name_key: "石斧"
    description_key: "最基础的伐木工具"
    definition_order: 100
    base_rarity: "Common"
    base_max_stack: 1
    sell_price: 0
    tags: ["Tool", "Axe"]
    tool_bonuses:
      log_yield_multiplier: 1.0
      chop_speed_multiplier: 1.0
```

你最常用的字段是：

- `id`
- `parent_id`
- `name_key`
- `description_key`
- `definition_order`
- `base_rarity`
- `base_max_stack`
- `sell_price`
- `tags`
- `tool_bonuses`

### 5.1 `tags` 的作用

`tags` 会影响很多系统判断，例如：

- 是否是工具
- 是否是斧头
- 是否能被某些技能识别

如果你要做伐木工具，通常至少要带：

```yaml
tags: ["Tool", "Axe"]
```

### 5.2 `tool_bonuses` 的作用

目前伐木类工具最关键的是：

- `log_yield_multiplier`
- `chop_speed_multiplier`

它们会影响挂机产量和速度。

## 6. 如何配置技能 `skills.yaml`

示例：

```yaml
skills:
  - id: "skill_chopping"
    name_key: "砍树"
    description_key: "伐木技能，等级越高效率越高"
    max_level: 20
    initial_level: 1
    max_total_exp: 34200
    required_tool_tag: "Axe"
    primary_output_item_id: "mat_wood_log"
    level_table:
      - { level: 1, exp_to_next: 180, output: 5.000, interval: 6.000 }
```

字段说明：

- `id`：技能唯一 ID
- `name_key`：技能显示名
- `description_key`：技能说明
- `max_level`：最高等级
- `initial_level`：初始等级
- `max_total_exp`：总经验上限
- `required_tool_tag`：需要的工具标签
- `primary_output_item_id`：挂机主要产出物
- `level_table`：等级成长表

## 7. 如何配置事件

当前事件仍然分成三种文件：

- `oneshot_events.yaml`：一次性事件
- `click_events.yaml`：单次点击事件
- `idle_events.yaml`：挂机事件

### 7.1 一次性事件示例

```yaml
events:
  - id: "evt_pickup_stone_axe"
    name_key: "捡起石斧"
    description_key: "你发现了一把石斧"
    hover_info_key: "获得石斧"
    type: "OneshotClick"
    prerequisites: []
    costs: []
    effects:
      - { effect_type: "GrantItem", target_id: "tool_stone_axe", int_value: 1 }
    button_list_group: "MainOneshot"
```

### 7.2 单次点击事件示例

```yaml
events:
  - id: "evt_click_chop"
    name_key: "点击砍树"
    description_key: "手动砍一下树"
    hover_info_key: "获得原木"
    type: "RepeatableClick"
    prerequisites:
      - { condition_type: "HasItem", target_id: "tool_stone_axe", required_value: 1 }
    costs: []
    effects:
      - { effect_type: "GrantItem", target_id: "mat_wood_log", int_value: 1 }
    button_list_group: "MainClick"
```

### 7.3 挂机事件示例

```yaml
events:
  - id: "evt_idle_chop"
    name_key: "挂机砍树"
    description_key: "自动持续砍树"
    hover_info_key: "持续获得原木与砍树经验"
    type: "IdleLoop"
    prerequisites:
      - { condition_type: "HasItem", target_id: "tool_stone_axe", required_value: 1 }
    linked_skill_id: "skill_chopping"
    button_list_group: "MainIdle"
```

### 7.4 事件常用字段说明

- `id`：事件唯一 ID
- `name_key`：按钮显示名称
- `description_key`：事件说明
- `hover_info_key`：鼠标悬浮说明
- `type`：事件类型
- `prerequisites`：前置条件
- `costs`：消耗
- `effects`：效果
- `linked_skill_id`：挂机事件关联技能
- `button_list_group`：旧按钮分组语义字段
- `remove_after_triggered`：触发后是否移除

### 7.5 当前新 UI 下必须特别记住的规则

这是这一版最容易搞混的地方。

在旧 UI 里，`button_list_group` 更像“显示在哪一列按钮里”。

但在当前新 UI 里，真正决定事件显示位置的，不再是 `button_list_group`，而是：

- `Configs/Scenarios/对应剧本/UI/gameplay_layout.yaml`
- 里面某个 `scene.event_ids`

也就是说：

- 事件 YAML 负责定义“事件本身是什么”
- `gameplay_layout.yaml` 负责定义“事件出现在哪个子场景里”

如果你只写了事件 YAML，但没有把事件 ID 填进某个子场景的 `event_ids`，那它不会自动显示出来。

## 8. `Zones/zones.yaml` 和 `UI/gameplay_layout.yaml` 到底分别记录什么

这个问题必须明确区分。

### 8.1 `zones.yaml` 记录的是玩法层的区域定义

它负责的是：

- 区域/副本 ID
- 区域说明
- 最大通关次数
- 解锁哪个新区域
- 解锁条件

它更偏向：

- 探索进度
- 清关次数
- 解锁链
- 战斗/副本层规则

也就是说，`zones.yaml` 是“系统规则层的区域”。

### 8.2 `gameplay_layout.yaml` 记录的是新 UI 的内容摆放结构

它负责的是：

- 一级区域
- 二级区域
- 子场景
- 每个子场景里显示哪些事件按钮

它更偏向：

- 左侧区域树怎么显示
- 中部当前区域页里有哪些子场景卡片
- 某个事件按钮应该出现在哪个子场景里

也就是说，`gameplay_layout.yaml` 是“界面布局层的区域”。

### 8.3 为什么两个文件里可能出现相近名称

例如“人间之里”“商店街”“魔法之森”这些名字，看起来像区域名，容易让人以为应该全部进 `zones.yaml`。

但实际上：

- 如果你是在定义“探索/战斗/清关/解锁逻辑”，放到 `zones.yaml`
- 如果你是在定义“左栏层级、二级区域、子场景、事件挂载位置”，放到 `gameplay_layout.yaml`

所以当前把“人间之里 / 商店街 / 琉璃点心屋 / 佐佐木木匠屋”这类新 UI 层级写到 `gameplay_layout.yaml`，不是放错了，而是职责不同。

## 9. 如何把事件绑定到新 UI 的子场景里

这是新 UI 下最关键的配置步骤之一。

测试剧本当前的“商店街”示例可以这样理解：

```yaml
areas:
  - id: "market_street"
    title: "商店街"
    accent: "secondary_peace"
    scenes:
      - id: "dessert_house"
        title: "琉璃点心屋"
        description: "人间之里里最热闹的小店之一。"
        event_ids: []

      - id: "tea_house"
        title: "甘味茶屋"
        description: "适合放慢节奏的休息点。"
        event_ids: []

      - id: "sasaki_carpenter_shop"
        title: "佐佐木木匠屋"
        description: "村里的木匠作坊，适合放置伐木入门、教程提示与木工相关事件。"
        event_ids:
          - "evt_pickup_stone_axe"
          - "evt_read_chopping_notice"
          - "evt_stranger_reward_offer"
```

这表示：

- “琉璃点心屋”当前没有事件按钮
- “甘味茶屋”当前没有事件按钮
- “捡起石斧”“阅读伐木心得”“路人的谢礼”只会出现在“佐佐木木匠屋”

### 9.1 为了避免事件误挂到多个场景，这一版新增了保护

现在 `GameplayUiConfigLoader` 在读取 `gameplay_layout.yaml` 时，会对重复绑定做检查：

- 如果同一个事件 ID 被写到了两个子场景里
- 后面的重复绑定会被忽略
- 同时在日志里给出警告

这样做的目的就是避免：

- 某个事件被你不小心同时挂到多个地方
- 最后在多个不该出现的场景里一起冒出来

## 10. 如何配置事件前置条件 `prerequisites`

示例：

```yaml
prerequisites:
  - { condition_type: "HasItem", target_id: "tool_stone_axe", required_value: 1 }
  - { condition_type: "SkillLevel", target_id: "skill_chopping", required_value: 2 }
```

当前常见条件包括：

- `HasItem`
- `HasGold`
- `SkillLevel`
- `FactionReputation`
- `ZoneCleared`
- `EventCompleted`
- `QuestCompleted`

## 11. 如何配置事件效果 `effects`

示例：

```yaml
effects:
  - { effect_type: "GrantItem", target_id: "tool_stone_axe", int_value: 1 }
  - { effect_type: "GrantGold", target_id: "", int_value: 30 }
  - { effect_type: "GrantSkillExp", target_id: "skill_chopping", int_value: 20 }
```

当前常见效果包括：

- `GrantItem`
- `RemoveItem`
- `GrantGold`
- `RemoveGold`
- `GrantSkillExp`
- `AddFactionReputation`
- `UnlockZone`
- `CompleteEvent`
- `CompleteQuest`

## 12. 如何配置“一次性事件弹窗”

当前仍然支持两种模式，而且在新 UI 下依然适用。

### 12.1 单确认按钮模式

```yaml
dialog:
  body_text: "你终于拿到了第一把石斧。"
  confirm_button_text: "我知道了"
```

这类弹窗现在仍会自动带一个：

- `我再想想`

玩家点这个按钮只会关闭弹窗，不会消耗这条一次性事件。

### 12.2 双选项分支模式

```yaml
dialog:
  body_text: "路人想送你谢礼。"
  consume_source_event_on_choice: true
  choices:
    - { button_text: "接受馈赠", target_event_id: "evt_accept_gift_gold" }
    - { button_text: "婉拒谢礼", target_event_id: "evt_decline_reward_food" }
```

推荐写法仍然是：

- 1 个源弹窗事件
- 2 个隐藏目标事件

目标事件通常建议：

- `button_list_group: "None"`

并且不要把目标事件再挂到 `gameplay_layout.yaml` 的 `event_ids` 里，否则它们会直接显示成普通按钮。

### 12.3 新 UI 下弹窗事件还要多记一条

在当前版本里，一次性事件要“能看见、能点到”，通常必须同时完成两步：

1. 在 `Configs/Scenarios/对应剧本/Events/oneshot_events.yaml` 里定义事件
2. 在 `Configs/Scenarios/对应剧本/UI/gameplay_layout.yaml` 里把事件 ID 挂到某个子场景的 `event_ids`

只做第一步，不做第二步，事件不会自动显示。

## 13. 如何配置势力与 NPC `factions.yaml`

示例：

```yaml
factions:
  - id: "faction_scarlet_devil"
    name_key: "红魔馆"
    description_key: "一个势力示例"
    max_reputation: 100
    peace_threshold: 100
    npc_ids: ["npc_sakuya"]
    faction_event_ids: []

npcs:
  - id: "npc_sakuya"
    name_key: "十六夜咲夜"
    faction_id: "faction_scarlet_devil"
    has_shop: true
    required_reputation: 20
    shop_items:
      - item_id: "tool_iron_axe"
        stock: 1
        payment_type: "Gold"
        gold_cost: 500
        required_reputation: 30
```

这里你可以配置：

- 势力名称
- 势力最大声望
- 势力下有哪些 NPC
- NPC 商店卖什么
- 购买需要多少金币或声望

## 14. 如何配置区域 `zones.yaml`

示例：

```yaml
zones:
  - id: "zone_forest_of_magic"
    name_key: "魔法之森"
    description_key: "区域示例"
    max_clear_count: 10
    unlocks_zone_id: "zone_misty_lake"
    unlocks_faction_id: ""
    unlock_conditions: []
```

字段含义：

- `id`：区域唯一 ID
- `name_key`：区域名称
- `description_key`：区域说明
- `max_clear_count`：最大通关计数
- `unlocks_zone_id`：解锁哪个下一区域
- `unlocks_faction_id`：解锁哪个势力
- `unlock_conditions`：解锁条件

再次强调：

- `zones.yaml` 不是拿来摆新 UI 左侧树结构的
- 新 UI 树结构、子场景和事件归属，要写在 `gameplay_layout.yaml`

## 15. 如何配置本地化文本 `locale_zh.yaml`

示例：

```yaml
locale: "zh"
translations:
  "石斧": "石斧"
  "砍树": "砍树"
  "挂机砍树": "挂机砍树"
```

当前为了方便你快速搭内容，很多地方也支持先直接写中文到：

- `name_key`
- `description_key`
- `body_text`

所以你可以先把流程跑通，后面要做正式多语言时再整理成统一 key。

## 16. 关于“货币”目前的真实情况

当前正式跑通的主货币仍然是：

- 金币 / Gold

你现在可以通过 YAML 配这些与金币相关的内容：

- 条件 `HasGold`
- 效果 `GrantGold / RemoveGold`
- 商店 `payment_type: Gold`
- 商店 `gold_cost`

但当前项目还没有做成完整的多货币 YAML 表，也就是还没有：

- `currencies.yaml`
- 多货币注册表
- 多货币 UI 展示
- 多货币统一消费规则

所以当前结论仍然是：

- 金币已经能正常配
- 新增一套完全独立的新货币并纯靠 YAML 让全系统自动识别，这一层还没完整落地

## 17. 关于“经验值”目前怎么配

### 17.1 技能升级经验

通过 `skills.yaml` 里的 `level_table.exp_to_next` 配。

### 17.2 事件直接给予经验

通过事件效果：

```yaml
- { effect_type: "GrantSkillExp", target_id: "skill_chopping", int_value: 20 }
```

### 17.3 挂机每轮经验

当前挂机经验仍然来自 `IdleSystem.cs` 的结算逻辑。

也就是说：

- `level_table.interval`
- `level_table.output`

不仅影响挂机效率，也会间接影响挂机经验节奏。

## 18. 当前最适合你的配置工作流

如果你继续扩写新剧本，我建议按这个顺序做：

1. 先复制一个剧本 `.tres`
2. 先复制一整套 `Configs/Scenarios/...` 子目录
3. 先补 `Localization/locale_zh.yaml`
4. 再补 `categories.yaml` 和 `items.yaml`
5. 再补 `skills.yaml`
6. 再补 `oneshot/click/idle_events.yaml`
7. 再补 `factions.yaml` 和 `zones.yaml`
8. 最后补 `UI/info_panel.yaml` 和 `UI/gameplay_layout.yaml`

这样最不容易乱。

## 19. 给你的一个非常实用的建议

因为你现在是“主线剧本”和“测试剧本”并行开发，我仍然建议始终遵守这一条：

- 新内容先在 `Configs/Scenarios/TestGame/` 里试
- 确认玩法没问题后，再人工搬到 `Configs/Scenarios/MainStory/`

在当前新 UI 版本里，这条建议还要多补一层：

- 左侧区域树结构、子场景划分、事件挂载位置，也先在 `Configs/Scenarios/TestGame/UI/gameplay_layout.yaml` 里试

这样能最大限度避免：

- 测试事件误混进正式主线
- 正式剧本的场景布局被试验性结构污染

## 20. 最后用一句话概括当前配置思路

现在最理想的理解方式是：

- `.tres` 决定载入哪套剧本
- 各种内容 YAML 决定这套剧本里有什么
- `gameplay_layout.yaml` 决定这些内容在新 UI 里摆到哪里
- 系统脚本决定这些内容按什么规则运行

你后面如果只是：

- 扩剧情
- 加物品
- 加事件
- 加区域
- 加子场景
- 给某个子场景挂事件

优先改 YAML 就对了。
## 21. 新 UI 下如何给一级区域、二级区域、子场景追加显示前置条件

现在 `Configs/Scenarios/对应剧本/UI/gameplay_layout.yaml` 里的这三层结构都支持显隐条件：

- 一级区域 `regions`
- 二级区域 `regions[].areas`
- 子场景 `regions[].areas[].scenes`

每一层都统一使用这两个字段：

- `visibility_condition_count`
- `visibility_conditions`

推荐写法如下：

```yaml
- id: "bamboo_forest"
  title: "迷途竹林"
  visibility_condition_count: 0
  visibility_conditions: []
  areas:
    - id: "eientei"
      title: "永远亭"
      visibility_condition_count: 1
      visibility_conditions:
        # TODO: 该事件尚未实例化，后续补事件定义。
        - { condition_type: "EventCompleted", target_id: "evt_talk_to_rabbit_chief", required_value: 1 }
      scenes:
        - id: "eientei_courtyard"
          title: "前庭"
          visibility_condition_count: 0
          visibility_conditions: []
          event_ids: []
```

这里要特别记住：

- `visibility_condition_count` 写的是“你声明自己配置了几条条件”
- `visibility_conditions` 写的才是“实际条件列表”

程序现在会检查这两者是否一致。

如果你写成：

- `visibility_condition_count: 2`
- 但 `visibility_conditions` 里只写了 1 条

那么加载时会打印错误提示，方便你排查 YAML 配置错误。

## 22. `visibility_conditions` 里每一条怎么写

每一条条件的结构和事件前置条件 `prerequisites` 是同一套格式：

```yaml
- { condition_type: "HasItem", target_id: "tool_stone_axe", required_value: 1 }
```

含义分别是：

- `condition_type`：条件类型
- `target_id`：要检查的目标 ID
- `required_value`：要求的数量/等级/次数

当前这套通用条件判定器已经支持：

- `None`
- `HasItem`
- `HasGold`
- `SkillLevel`
- `FactionReputation`
- `ZoneCleared`
- `EventCompleted`
- `QuestCompleted`

最常用的几种写法示例如下：

```yaml
visibility_conditions:
  - { condition_type: "HasItem", target_id: "item_moon_key", required_value: 1 }
```

表示：

- 背包里至少要有 1 个 `item_moon_key`

```yaml
visibility_conditions:
  - { condition_type: "EventCompleted", target_id: "evt_help_reimu", required_value: 1 }
```

表示：

- 必须完成过一次性事件 `evt_help_reimu`

```yaml
visibility_conditions:
  - { condition_type: "ZoneCleared", target_id: "zone_bamboo_trial", required_value: 1 }
```

表示：

- 必须至少通关过一次 `zone_bamboo_trial`

如果你要写多条件，就继续往下追加多条，当前逻辑是：

- 所有条件同时满足才显示

例如：

```yaml
visibility_condition_count: 2
visibility_conditions:
  - { condition_type: "EventCompleted", target_id: "evt_help_reimu", required_value: 1 }
  - { condition_type: "HasItem", target_id: "item_moon_key", required_value: 1 }
```

这表示两个条件缺一不可。

## 23. 父级隐藏时，子级不会越级显示

现在新 UI 的显示逻辑已经明确做成了“父子联动”：

- 如果一级区域不可见，那么它下面的二级区域不会显示
- 如果二级区域不可见，那么它下面的子场景不会显示

即使某个子级自己的 `visibility_conditions` 已经满足，只要它的父级还没显示，它也不会单独跳出来。

这样做的目的是让界面更符合直觉，避免出现：

- 左边一级区域没开
- 但中间却莫名其妙出现下属子场景

另外当前默认选中区域也做了保护：

- 如果原本存档里记住的区域现在变成不可见
- UI 会自动回退到“当前第一个可见的区域”

所以不会因为读档后某个区域隐藏而直接把界面卡死。

## 24. 这些显示条件依赖的历史数据会不会丢存档

不会。

这次区域显隐条件复用的是玩家档里已经存在、并且已经会保存的历史数据：

- 背包道具：来自 `PlayerProfile.Inventory`
- 已完成一次性事件：来自 `PlayerProfile.CompletedEventIds`
- 已完成任务：来自 `PlayerProfile.CompletedQuestIds`
- 区域通关次数：来自 `PlayerProfile.ZoneStates[*].ClearCount`

也就是说：

- 玩家拿到过解锁道具
- 玩家做完过解锁事件
- 玩家打通过某个副本

这些状态都会跟着存档一起保存。

所以读档后不会出现“明明以前已经解锁过，但因为历史信息没保存导致区域重新隐藏”的卡关问题。

## 27. 现在事件按钮也拆成了三套条件

这次开始，`一次性事件 / 单次点击事件 / 循环挂机事件` 三类按钮事件，都可以分别配置：

- 前置显示条件
- 前置互动条件
- 后置隐藏条件

对应 YAML 字段分别是：

- `display_condition_count`
- `display_conditions`
- `interaction_condition_count`
- `interaction_conditions`
- `hide_condition_count`
- `hide_conditions`

这三套条件的职责要严格区分：

### 27.1 `display_conditions`

控制：

- 这个按钮要不要出现在界面上

只要显示条件不满足，就直接隐藏按钮，不给玩家看到。

### 27.2 `interaction_conditions`

控制：

- 按钮已经显示出来后，当前能不能点

如果显示条件满足，但互动条件还不满足，那么按钮会：

- 显示在界面上
- 但处于置灰不可点击状态

这就是现在新 UI 下“显示但不可互动”的正式配置入口。

### 27.3 `hide_conditions`

控制：

- 事件在后续某些条件满足后，是否应该彻底隐藏

它可以用来表达这类情况：

- 原本显示且可点击，后来变成隐藏
- 原本显示但置灰，后来直接隐藏
- 原本显示且可点击，后续因为剧情推进或状态变化而不再显示

## 28. 这三套条件怎么写

每一条条件仍然沿用统一条件结构：

```yaml
- { condition_type: "HasItem", target_id: "tool_fishing_rod", required_value: 1 }
```

含义是：

- `condition_type`：条件类型
- `target_id`：目标 ID
- `required_value`：数量 / 等级 / 次数

当前按钮事件和区域显隐共用同一套条件判定器，所以支持的条件类型也是同一批：

- `None`
- `HasItem`
- `HasGold`
- `SkillLevel`
- `FactionReputation`
- `ZoneCleared`
- `EventCompleted`
- `QuestCompleted`

例如：

```yaml
display_condition_count: 2
display_conditions:
  - { condition_type: "HasItem", target_id: "tool_fishing_rod", required_value: 1 }
  - { condition_type: "EventCompleted", target_id: "evt_talk_to_taigongwang", required_value: 1 }
interaction_condition_count: 0
interaction_conditions: []
hide_condition_count: 0
hide_conditions: []
```

表示：

- 只有在“持有鱼竿”并且“完成与太公望对话”后，按钮才会显示
- 一旦显示出来，就可以直接点击
- 当前没有额外的后置隐藏条件

## 29. 数量字段现在会做一致性校验

无论是：

- `display_condition_count`
- `interaction_condition_count`
- `hide_condition_count`

程序都会检查它和对应条件列表的实际条数是否一致。

例如你写成：

```yaml
interaction_condition_count: 2
interaction_conditions:
  - { condition_type: "HasItem", target_id: "tool_fishing_rod", required_value: 1 }
```

那加载时就会报错并打印提示，因为你声明了 2 条互动条件，但实际只写了 1 条。

这样做的目的就是避免：

- 后期自己改 YAML 时少写一条
- 或复制粘贴后忘记同步修改数量
- 最终出现按钮显隐逻辑异常却不好排查

## 30. 旧字段 `prerequisites` 现在怎么处理

为了兼容老配置，事件 YAML 里的旧字段：

- `prerequisites`

现在会被当作：

- `interaction_conditions`

也就是说：

- 老剧本暂时不改，也还能继续跑
- 但新配置建议以后统一改写成 `interaction_condition_count + interaction_conditions`

这样表达会更清晰，因为它明确说明了：

- 这是控制“按钮能不能点”
- 而不是控制“按钮显不显示”

## 31. 新逻辑下按钮会有三种界面状态

对于一个事件按钮，现在会出现这三种状态：

### 31.1 隐藏

出现条件：

- 显示条件不满足
- 或隐藏条件满足
- 或一次性事件已经被消耗

### 31.2 显示但置灰

出现条件：

- 显示条件满足
- 但互动条件不满足

这类按钮会显示出来，但玩家不能点击。

### 31.3 显示且可点击

出现条件：

- 显示条件满足
- 互动条件满足
- 也没有命中隐藏条件

这时按钮会正常高亮，允许玩家点击。

## 32. 挂机事件现在支持“每轮消耗成本”

这次专门补上了一个以前没有真正接完的逻辑：

- `IdleLoop` 挂机事件现在会读取 `costs`
- 并在每轮结算时扣除对应成本

也就是说，像“挂机钓鱼”这种事件，现在可以写成：

```yaml
costs:
  - { item_id: "bait_worm", amount: 1 }
```

这表示：

- 每完成一轮挂机钓鱼，就消耗 1 个 `bait_worm`

如果成本不够了，系统会：

- 自动停止挂机
- 清空当前挂机事件记录
- 刷新按钮状态

所以你后面可以继续用这套接口做：

- 钓鱼消耗鱼饵
- 炼药消耗药材
- 烹饪消耗食材
- 锻造消耗矿石

## 34. 两个钓鱼例子分别怎么配

### 34.1 `evt_idle_fishing` 挂机钓鱼

这个例子演示的是：

- 按钮一开始就显示
- 但只有满足互动条件后才点亮

当前写法是：

```yaml
type: "IdleLoop"
display_condition_count: 0
display_conditions: []
interaction_condition_count: 2
interaction_conditions:
  - { condition_type: "HasItem", target_id: "tool_fishing_rod", required_value: 1 }
  - { condition_type: "HasItem", target_id: "bait_worm", required_value: 1 }
hide_condition_count: 0
hide_conditions: []
costs:
  - { item_id: "bait_worm", amount: 1 }
linked_skill_id: "skill_fishing"
```

它表达的含义是：

- 从开局开始玩家就能看见“挂机钓鱼”按钮
- 如果没有鱼竿和鱼饵，它会显示但置灰
- 同时持有鱼竿和鱼饵后，它会点亮
- 每轮结算消耗 1 个鱼饵
- 鱼饵耗尽后，挂机自动停止

### 34.2 `evt_empty_hook_fishing` 姜太公钓鱼

这个例子演示的是：

- 满足显示条件前，按钮根本不出现

```yaml
type: "RepeatableClick"
display_condition_count: 2
display_conditions:
  - { condition_type: "HasItem", target_id: "tool_fishing_rod", required_value: 1 }
  - { condition_type: "EventCompleted", target_id: "evt_talk_to_taigongwang", required_value: 1 }
interaction_condition_count: 0
interaction_conditions: []

```

它表达的含义是：

- 没鱼竿时按钮不显示
- 没完成“与太公望对话”时按钮也不显示
- 两个条件都满足后，按钮才会真正出现在界面中

## 35. “与太公望对话”为什么要进存档

因为像：

- `evt_talk_to_taigongwang`

这种一次性事件，一旦完成，就会影响后续按钮是否显示。

比如这次的：

- `evt_empty_hook_fishing`

就依赖：

- `EventCompleted: evt_talk_to_taigongwang`

所以“已完成的一次性事件历史”必须进存档。

当前这部分数据已经保存在：

- `PlayerProfile.CompletedEventIds`

并且 `SaveManager` 在存档里会持久化这组数据。

所以玩家读档后不会出现：

- 明明以前已经和太公望对过话
- 结果系统忘了这条历史
- 导致“姜太公钓鱼”按钮重新消失

## 36. 以后该怎么选用这三套条件

最简单的经验法则是：

如果你想表达：

- “玩家现在还不该看到这个事件”

就写：

- `display_conditions`

如果你想表达：

- “玩家可以先看到，但还不能点”

就写：

- `interaction_conditions`

如果你想表达：

- “玩家推进到某一步后，这个按钮应该彻底消失”

就写：

- `hide_conditions`

这样配置出来的语义最清楚，也最不容易把显示逻辑和互动逻辑混在一起。
