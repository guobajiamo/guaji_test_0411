# 30_新UI版本_项目运行时调用关系与讲解文档

## 1. 先看最短结论

当前项目运行时主链路，可以先记成这一条：

```text
Main.tscn
  -> AppRoot.cs
    -> MainMenuUI.cs
      -> GameManager.StartScenario(...)
        -> 读取 .tres 剧本资源
        -> 读取该剧本指向的 YAML
        -> 注册物品/技能/事件/势力/区域
        -> 读取新 UI 主题与 gameplay_layout.yaml
        -> 初始化 PlayerProfile 与各系统
          -> GamePlayUI.cs
            -> 顶部剧本/区域栏
            -> 左侧多级区域树
            -> 中部当前区域/背包/技能/图鉴/系统
            -> 中部底部日志
            -> 右侧信息面板与状态栏
```

如果你后面遇到问题，基本还是沿着这条链往下排查。

## 2. 启动阶段是怎么走的

### 2.1 Godot 启动入口

启动时先读取：

- `project.godot`

这里决定：

- 主场景是 `Scenes/Main.tscn`
- 自动加载 `GameManager`
- 自动加载 `SignalBus`
- 设计基准分辨率仍然是 `1920x1080`

### 2.2 自动加载单例先准备好

在真正进入主场景之前，Godot 会先初始化：

- `Scripts/Autoload/GameManager.cs`
- `Scripts/Autoload/SignalBus.cs`

其中真正关键的是 `GameManager`。

它在 `_Ready()` 里会做这些事：

1. 扫描 `Resources/Scenarios/` 下所有 `.tres` 剧本资源
2. 建立剧本列表
3. 创建默认空玩家档案 `PlayerProfile`
4. 初始化各玩法系统节点
5. 记录一条“主菜单入口已准备完成”的日志

这一步结束后，程序已经知道：

- 有哪些剧本可用
- 点击“新的游戏”会进哪个默认剧本
- 点击“进入游戏测试”会进哪个测试剧本

## 3. 主场景 `Main.tscn` 做什么

`Scenes/Main.tscn` 现在仍然只是一个轻量壳。

它挂载的脚本是：

- `Scripts/UI/AppRoot.cs`

`AppRoot` 的职责仍然是：

- 在主菜单和游戏内界面之间切换
- 读取窗口布局参数
- 管理窗口缩放
- 接收主菜单按钮的跳转请求

## 4. 主菜单 `MainMenuUI.cs` 怎么工作

主菜单脚本路径：

- `Scripts/UI/MainMenuUI.cs`

它当前主要负责：

- 显示主菜单背景
- 创建“新的游戏 / 读取存档 / 设置 / 退出”按钮
- 提供右上角“进入游戏测试”按钮

它不自己决定读哪套剧本，而是把选择交回：

- `AppRoot.StartDefaultStory()`
- `AppRoot.OpenTestGame()`
- `AppRoot.LoadStoryFromMenu(path)`

## 5. `GameManager.StartScenario()` 为什么仍然是核心

不管是正式剧本还是测试剧本，最后都会走到：

- `GameManager.StartScenario(string scenarioId, string savePath, bool resetProfile)`

它负责：

1. 根据 `scenarioId` 找到对应 `GameScenarioDefinition`
2. 设置 `ActiveScenario`
3. 视情况重置玩家档案
4. 设置当前存档路径
5. 调用 `LoadStaticData(scenario)` 读取 Items / Skills / Events / Factions / Zones / Localization
6. 调用 `InitializeProfileFromDefinitions()` 给运行时档案补齐默认状态
7. 调用 `ConfigureSystems()` 让各系统接入注册表
8. 写入“已载入剧本”日志

## 6. 剧本静态数据怎么读进来

`GameManager.LoadStaticData(scenario)` 会根据 `.tres` 里的路径去读：

- `CategoriesConfigPath`
- `ItemsConfigPath`
- `SkillsConfigPath`
- `OneshotEventsConfigPath`
- `ClickEventsConfigPath`
- `IdleEventsConfigPath`
- `FactionsConfigPath`
- `ZonesConfigPath`
- `LocalizationPath`

读取者是：

- `Scripts/Core/Helpers/YamlConfigLoader.cs`

然后分别装入：

- `ItemRegistry`
- `SkillRegistry`
- `EventRegistry`
- `FactionRegistry`
- `ZoneRegistry`
- `LocalizationManager`

## 7. 新 UI 配置是怎么读进来的

这轮新增的一层，是新 UI 自己也有一套配置读取链路。

当前流程是：

```text
GameScenarioDefinition.InfoPanelConfigPath
  -> GameplayUiConfigLoader.ResolveScenarioLayoutPath(...)
    -> 同目录 gameplay_layout.yaml
```

也就是说：

- 剧本资源里目前没有新增一个单独的 `GameplayLayoutPath`
- 程序会根据 `InfoPanelConfigPath` 所在目录
- 自动去找同目录下的 `gameplay_layout.yaml`

与此同时，所有剧本共用一份：

- `Configs/UI/gameplay_ui_theme.yaml`

这份文件负责控制新 UI 的整体配色、透明度、字号、区域宽度等。

## 8. 游戏内主界面 `GamePlayUI` 现在怎么工作

剧本加载成功后，`AppRoot` 会切换到：

- `Scenes/UI/MainUI.tscn`

这个场景本身仍只是壳。

真正逻辑现在拆成三份：

- `Scripts/UI/GamePlayUI.cs`
- `Scripts/UI/GamePlayUI.Layout.cs`
- `Scripts/UI/GamePlayUI.Actions.cs`

你可以把它理解为：

- `GamePlayUI.cs` 管主骨架与刷新入口
- `GamePlayUI.Layout.cs` 管布局、左侧区域树、中部页签、场景卡片装配
- `GamePlayUI.Actions.cs` 管按钮行为、弹窗、存读档、返回主菜单、事件执行

## 9. 新 UI 运行时是怎么装配出来的

### 9.1 顶部栏

顶部栏会显示：

- 当前剧本名称
- 当前选中的二级区域名称

### 9.2 左侧区域树

左侧会根据 `gameplay_layout.yaml` 生成：

- 一级地区
- 一级地区展开/折叠按钮
- 二级区域按钮
- 二级区域右侧绿色“可互动按钮数”
- 当前挂机所在区域左侧红色 `★`

### 9.3 中部标签页

当前中部固定生成这些页：

- 当前区域
- 背包
- 技能
- 图鉴
- 系统

其中：

- “当前区域”页不是旧版那种单独的区域标签页
- 而是根据左侧当前选中的二级区域，动态显示该区域下的所有子场景卡片

### 9.4 当前区域页里的子场景卡片

每个子场景都会生成一张独立卡片。

每张卡片里会显示：

- 场景标题
- 场景描述
- 该场景下的事件按钮

事件按钮来源于：

- 对应 `SceneLayout.event_ids`

也就是说：

- 新 UI 下，事件显示在哪个场景里
- 是由 `gameplay_layout.yaml` 控制的

### 9.5 事件排序逻辑

场景内事件按钮按下面顺序排序：

1. 一次性事件 `OneshotClick`
2. 单次点击事件 `RepeatableClick`
3. 挂机事件 `IdleLoop`

按钮颜色也按事件类型区分：

- 一次性事件：红色
- 单次点击事件：绿色
- 挂机事件：橙色

### 9.6 场景收藏

在当前区域页里：

- 右键场景标题可以切换收藏
- 被收藏的场景会置顶
- 如果收藏多个，最新收藏的排在最前面

这个顺序记录在：

- `PlayerUiState.FavoriteSceneIds`

### 9.7 系统页

现在正式剧本和测试剧本都启用了系统页。

区别是：

- 正式剧本系统页：
  - 槽位保存
  - 槽位读取
  - 返回主菜单

- 测试剧本系统页：
  - 一键存档
  - 一键读档
  - 返回主菜单

也就是说，本轮已经把测试剧本原本底部那组快捷存读档入口，收口进统一系统页。

## 10. 左侧区域点击后会发生什么

当玩家点击左侧某个二级区域时，`GamePlayUI` 会做这些事：

1. 记录当前区域到 `PlayerUiState.SelectedAreaId`
2. 自动把中部标签切回“当前区域”
3. 重新刷新顶部栏
4. 重新刷新左侧区域树
5. 重新刷新“当前区域”页里的子场景卡片

所以这一步之后，玩家会立刻看到：

- 顶部的当前区域名变了
- 中部当前区域页内容切换到对应区域

## 11. 一次性事件弹窗在新 UI 下怎么流转

这套逻辑在新 UI 下仍然保留，没有因为布局改版而失效。

流程仍然是：

1. 场景卡片里的事件按钮被点击
2. `GamePlayUI.OnEventButtonPressed(eventId)` 找到事件定义
3. 如果事件带 `dialog`
4. 则走 `GamePlayUI.ShowDialogForEvent(...)`
5. 弹出 `EventDialogPanel`
6. 玩家确认或选择后，再去调用：
   - `ClickEventSystem.TryTriggerEvent(...)`
   - 或 `ClickEventSystem.TryTriggerDialogChoice(...)`

也就是说：

- 18 号文档里的“一次性事件弹窗两种情况配置方案”
- 在现在新 UI 下仍然适用

唯一需要更新的理解是：

- 旧文档里更偏向“左侧三列按钮”
- 现在这些一次性事件按钮是显示在“某个二级区域下的某个子场景卡片里”

## 12. 挂机事件现在怎么流转

挂机事件按钮点击后，流程仍然是：

- `GamePlayUI.HandleIdleEvent(...)`
  -> `IdleSystem.StartIdleEvent(eventId)`

之后真正推进发生在：

- `IdleSystem._Process(double delta)`

而新 UI 层多做了三件事：

1. 在挂机按钮上显示红色 `★`
2. 在挂机按钮上方显示进度条
3. 在左侧对应二级区域前显示红色 `★`

这样玩家即使切到背包或技能页，也能通过右侧目标栏和左侧区域树确认是否仍在正常挂机。

## 13. 新 UI 下存档系统怎么流转

底层 JSON 存档执行者仍然是：

- `Scripts/Core/SaveLoad/SaveManager.cs`

但这轮多保存了一层新 UI 运行态：

- 当前选中二级区域 `SelectedAreaId`
- 当前选中标签页 `SelectedTabId`
- 收藏场景顺序 `FavoriteSceneIds`

因此：

- 存档恢复后，UI 会更接近玩家离开前的状态

## 14. 当前最常见的排查顺序

如果你以后遇到“区域不对 / 场景不对 / 按钮不对 / 弹窗不对”，建议按这个顺序排查：

1. 看 `Resources/Scenarios/*.tres` 是否指向正确的剧本 YAML
2. 看对应剧本 `Configs/Scenarios/*/Events/*.yaml` 是否真的写了这些事件
3. 看对应剧本 `Configs/Scenarios/*/UI/gameplay_layout.yaml` 是否把这些事件挂到了正确场景
4. 看 `GamePlayUI.Layout.cs` 是否把对应区域和场景正确生成出来
5. 再看 `ClickEventSystem` / `IdleSystem` / `SkillSystem` 是否执行成功

## 15. 用一句话概括当前运行时架构

你可以把现在的架构理解成：

- `AppRoot` 管场景切换
- `GameManager` 管剧本加载和系统初始化
- `YamlConfigLoader` 管静态配置读取
- `GameplayUiConfigLoader` 管新 UI 主题与区域布局读取
- `Systems` 管玩法执行
- `GamePlayUI` 管新 UI 布局展示与交互
- `SaveManager` 管持久化

这就是当前“新 UI 版本”的完整运行时骨架。
