## 2. 当前真正生效的剧本配置目录

当前真正由剧本资源 `.tres` 指向并实际生效的，仍然是：

### 2.1 主线剧本配置目录

- `Configs/Scenarios/MainStory/Items/`
- `Configs/Scenarios/MainStory/Skills/`
- `Configs/Scenarios/MainStory/Events/`
- `Configs/Scenarios/MainStory/Factions/`
- `Configs/Scenarios/MainStory/Zones/`
- `Configs/Scenarios/MainStory/Localization/`
- `Configs/Scenarios/MainStory/UI/`

### 2.2 测试剧本配置目录

- `Configs/Scenarios/TestGame/Items/`
- `Configs/Scenarios/TestGame/Skills/`
- `Configs/Scenarios/TestGame/Events/`
- `Configs/Scenarios/TestGame/Factions/`
- `Configs/Scenarios/TestGame/Zones/`
- `Configs/Scenarios/TestGame/Localization/`
- `Configs/Scenarios/TestGame/UI/`

相比旧版本，这一轮新增最重要的是：

- 每个剧本现在除了事件、技能、物品配置外，还各自拥有一套 `UI/` 子目录
- `UI/` 里至少有两份配置：
  - `info_panel.yaml`
  - `gameplay_layout.yaml`

## 3. 主线剧本和测试剧本分别在哪里看

### 3.1 主线剧本资源文件

- `Resources/Scenarios/main_story_scenario.tres`

这个文件决定：

- 主线剧本的 `ScenarioId`
- 主线剧本显示名称
- 是否是默认“新的游戏”入口
- 主线剧本引用哪一套 Items / Skills / Events / Factions / Zones / Localization YAML
- 主线剧本右上角信息面板引用哪份 `info_panel.yaml`
- 是否启用“系统”标签页

### 3.2 测试剧本资源文件

- `Resources/Scenarios/test_game_scenario.tres`

这个文件决定：

- 测试剧本的 `ScenarioId`
- 是否是测试入口
- 测试剧本引用哪一套 YAML
- 测试剧本右上角信息面板引用哪份 `info_panel.yaml`
- 是否启用“系统”标签页

当前这版里，测试剧本也已经启用：

- `EnableSystemTab = true`

也就是说，测试剧本与正式剧本现在都共用系统页入口，只是按钮行为不同。

## 4. 当前新 UI 相关目录分别负责什么

### 4.1 `Configs/UI/gameplay_ui_theme.yaml`

这是新 UI 的全局主题配置。

它负责：

- 黑底背景色
- 顶部栏、左栏、中栏、底部日志、右上信息区、右下状态区的底色与透明度
- 左右栏宽度
- 区域字号
- 标签字号
- 场景卡片尺寸
- 进度条高度
- 目标/技能/状态/货币/声望栏标题文案

### 4.2 `Configs/Scenarios/*/UI/gameplay_layout.yaml`

这是每个剧本自己的“多级区域布局配置”。

它负责：

- 默认进入哪个二级区域
- 有哪些一级地区
- 每个一级地区下有哪些二级区域
- 每个二级区域下有哪些子场景
- 每个子场景里放哪些事件按钮

也就是说，在新 UI 里：

- “事件按钮显示在哪个场景中”
  不再靠旧的三列布局硬编码
- 而是由 `gameplay_layout.yaml` 决定

### 4.3 `Scripts/UI/GameplayUiConfig.cs`

这是新 UI 配置读取器与数据结构定义文件。

它负责把：

- `Configs/UI/gameplay_ui_theme.yaml`
- `Configs/Scenarios/*/UI/gameplay_layout.yaml`

读成程序里的对象，例如：

- `GameplayUiTheme`
- `ScenarioGameplayLayout`
- `PrimaryRegionLayout`
- `SecondaryAreaLayout`
- `SceneLayout`

### 4.4 `Scripts/Core/Runtime/PlayerUiState.cs`

这是新 UI 的玩家运行时状态。

它负责记录：

- 当前选中的二级区域
- 当前选中的中部标签页
- 场景收藏顺序

并且已经接入存档，因此：

- 读档后会恢复当前区域
- 读档后会恢复当前标签页
- 读档后会恢复场景收藏顺序

### 5.1 主菜单层

仍由：

- `Scenes/Main.tscn`
- `Scripts/UI/AppRoot.cs`

配合完成。

主菜单本轮没有改动。

### 5.2 游戏内主界面层

由 `Scenes/UI/MainUI.tscn` 挂壳，实际逻辑在：

- `Scripts/UI/GamePlayUI.cs`
- `Scripts/UI/GamePlayUI.Layout.cs`

当前这层主要负责：

- 顶部剧本/区域栏
- 左侧多级区域树
- 中部“当前区域 / 背包 / 技能 / 图鉴 / 系统”标签页
- 中部底部日志区
- 右上信息面板
- 右下人物状态栏
- 一次性事件弹窗
- 正式与测试两套存读档入口

### 5.3 中部标签页层

当前中部区域主要包括：

- 当前区域
- 背包
- 技能
- 图鉴
- 系统

已经移除的旧标签页：

- 势力
- 区域

## 6. 当前资源与配置之间的关系

程序真正运行时，关系可以理解成：

1. `.tres` 剧本资源决定“读哪套 Items / Skills / Events / Zones / UI 配置”
2. `YamlConfigLoader` 把 Items / Skills / Events / Zones / Localization 读成定义对象
3. `GameplayUiConfigLoader` 把新 UI 主题和区域布局 YAML 读成 UI 配置对象
4. `Registry` 把静态定义注册起来
5. `GameManager` 把注册表和当前活动剧本交给系统与 UI
6. `GamePlayUI` 根据 `gameplay_layout.yaml` 把事件显示到对应的二级区域和子场景里

所以你后面如果发现：

- 某个事件不该出现在某个子场景却出现了
- 某个区域点开后显示错了场景
- 某个剧本的默认区域不对

第一优先不是看 `ClickEventSystem`，而是先看：

- 对应剧本的 `UI/gameplay_layout.yaml`

## 8. 当前结构的一个实用理解方式

你可以把现在的项目理解成四层：

- 第一层：`Scenes/` 与 `Scripts/UI/` 负责“玩家看到什么”
- 第二层：`Scripts/Systems/` 负责“按钮按下后发生什么”
- 第三层：`Resources/Scenarios/` + `Configs/Scenarios/` 负责“这次剧本装什么内容”
- 第四层：`Configs/UI/` + `Configs/Scenarios/*/UI/` 负责“这些内容在新 UI 下怎么摆出来”

这样后面不管你是：

- 换 UI 配色
- 加一级地区
- 加二级区域
- 加子场景
- 改某个场景里显示哪些事件

都会更容易判断该去哪个目录动手。
