# 23_新剧本版本_项目结构说明文档

## 1. 项目结构总览

## 2. 当前真正生效的剧本配置目录

这是这轮最重要的变化之一。

当前真正生效的剧本配置，不再是旧的：

- `Configs/Items/`
- `Configs/Skills/`
- `Configs/Events/`
- `Configs/Factions/`
- `Configs/Zones/`
- `Configs/Localization/`

这些旧目录现在更适合视为“早期公共样例/旧版本残留参考”。

当前真正由剧本资源 `.tres` 指向并实际生效的，是：

### 2.1 主线剧本配置目录

- `Configs/Scenarios/MainStory/Items/`
- `Configs/Scenarios/MainStory/Skills/`
- `Configs/Scenarios/MainStory/Events/`
- `Configs/Scenarios/MainStory/Factions/`
- `Configs/Scenarios/MainStory/Zones/`
- `Configs/Scenarios/MainStory/Localization/`

### 2.2 测试剧本配置目录

- `Configs/Scenarios/TestGame/Items/`
- `Configs/Scenarios/TestGame/Skills/`
- `Configs/Scenarios/TestGame/Events/`
- `Configs/Scenarios/TestGame/Factions/`
- `Configs/Scenarios/TestGame/Zones/`
- `Configs/Scenarios/TestGame/Localization/`

## 3. 主线剧本和测试剧本分别在哪里看

### 3.1 主线剧本资源文件

- `Resources/Scenarios/main_story_scenario.tres`

这个文件决定：

- 主线剧本的 `ScenarioId`
- 主线剧本显示名称
- 是否是默认“新的游戏”入口
- 主线剧本引用哪一套 YAML
- 是否显示“系统”标签页

### 3.2 测试剧本资源文件

- `Resources/Scenarios/test_game_scenario.tres`

这个文件决定：

- 测试剧本的 `ScenarioId`
- 是否是测试入口
- 是否显示正式系统标签
- 测试剧本引用哪一套 YAML

## 4. 为什么以前“新的游戏”里也会出现石斧事件

因为之前主线剧本和测试剧本虽然名字不同，但底层都还在引用同一套配置。

也就是说：

- 主线剧本 `.tres` 指到了旧测试事件 YAML
- 测试剧本 `.tres` 也指到了同一套 YAML

结果就是：

- “新的游戏”进入主线剧本时，仍然会加载石斧事件链
- 表面上看像是主线/测试分开了
- 实际上底层没有真正分开

现在已经修正为：

- 主线剧本指向 `Configs/Scenarios/MainStory/...`
- 测试剧本指向 `Configs/Scenarios/TestGame/...`

其中：

- `MainStory` 当前默认是空配置
- `TestGame` 当前保留石斧、砍树、挂机、弹窗选择等测试链路

## 5. 当前各层目录分别负责什么

### 5.1 `Scenes/`

只负责“场景壳”和入口。

现在大部分 UI 结构是代码动态搭出来的，所以场景文件本身很轻。

### 5.2 `Scripts/Core/Definitions/`

负责静态数据结构定义，例如：

- `ItemDefinition.cs`
- `SkillDefinition.cs`
- `EventDefinition.cs`
- `FactionDefinition.cs`
- `ZoneDefinition.cs`

你可以把它理解成“YAML 一条条配置最终会装进什么样的 C# 对象里”。

### 5.3 `Scripts/Core/Registry/`

负责把已经加载好的定义按 ID 管起来，供系统查询。

例如：

- `ItemRegistry` 根据物品 ID 取物品
- `EventRegistry` 根据事件 ID 取事件
- `SkillRegistry` 根据技能 ID 取技能

### 5.4 `Scripts/Core/Runtime/`

负责玩家当前实际进度数据。

这里记录的是“玩到现在是什么状态”，不是“游戏本来定义了什么内容”。

例如：

- 背包里现在有多少原木
- 当前金币是多少
- 当前技能等级是多少
- 当前挂机是否正在进行
- 哪些一次性事件已经完成

### 5.5 `Scripts/Core/Helpers/YamlConfigLoader.cs`

这是 YAML 配置读取器。

它负责把：

- 物品 YAML
- 技能 YAML
- 事件 YAML
- 势力 YAML
- 区域 YAML
- 本地化 YAML
- UI 图片主题 YAML

读成程序里的对象。

### 5.6 `Scripts/Systems/`

负责具体玩法逻辑。

当前最关键的是：

- `ClickEventSystem.cs`
- `IdleSystem.cs`
- `SkillSystem.cs`

它们分别处理：

- 点击触发事件
- 挂机事件循环
- 技能经验与升级

### 5.7 `Scripts/UI/`

负责所有界面逻辑。

当前最关键的是：

- `AppRoot.cs`
- `MainMenuUI.cs`
- `GamePlayUI.cs`
- `SystemPanel.cs`
- `UiImageThemeManager.cs`

## 6. 当前 UI 结构怎么分层

### 6.1 主菜单层

由下面两部分配合：

- `Scenes/Main.tscn`
- `Scripts/UI/AppRoot.cs`

启动后先进入主菜单，再决定是进入：

- 正式主线剧本
- 测试剧本
- 读取正式存档

### 6.2 游戏内主界面层

由 `Scenes/UI/MainUI.tscn` 挂壳，然后实际逻辑在：

- `Scripts/UI/GamePlayUI.cs`

当前这层主要负责：

- 左侧三列事件按钮
- 中间标签页
- 右侧状态与提示
- 底部日志区
- 弹窗对话框
- 正式存读档
- 测试模式快捷存读档

### 6.3 标签页层

中间区域当前主要包括：

- 背包
- 技能
- 图鉴
- 势力
- 区域
- 系统（仅正式剧本显示）

## 7. 当前资源与配置之间的关系

程序真正运行时，关系是这样的：

1. `.tres` 剧本资源决定“这次要读哪套 YAML”
2. `YamlConfigLoader` 把 YAML 读成定义对象
3. `Registry` 把这些对象注册起来
4. `GameManager` 把这些注册表交给各个系统使用
5. `GamePlayUI` 根据当前注册表把按钮和标签页显示出来

所以你后面如果发现：

- 某个剧本不该有某个事件却有
- 某个剧本不该有某个物品却有

第一优先不是看 UI，而是先看对应剧本 `.tres` 到底指向了哪套 YAML。

## 8. 当前最值得你记住的几个路径

如果你后面自己查项目，最值得先记住的是这几个：

- 主菜单总入口：`Scenes/Main.tscn`
- 总控切换脚本：`Scripts/UI/AppRoot.cs`
- 游戏内主界面主体：`Scripts/UI/GamePlayUI.cs`
- 主线剧本资源：`Resources/Scenarios/main_story_scenario.tres`
- 测试剧本资源：`Resources/Scenarios/test_game_scenario.tres`
- 主线剧本 YAML 根目录：`Configs/Scenarios/MainStory/`
- 测试剧本 YAML 根目录：`Configs/Scenarios/TestGame/`
- UI 图片主题：`Configs/UI/ui_image_theme.yaml`
- UI 图片素材目录：`Assets/UI/Images/`

## 9. 当前结构的一个实用理解方式

你可以把现在的项目理解成三层：

- 第一层：`Scenes/` 和 `Scripts/UI/` 负责“玩家看到什么”
- 第二层：`Scripts/Systems/` 负责“按钮按下后发生什么”
- 第三层：`Resources/Scenarios/` + `Configs/Scenarios/` 负责“这次剧本到底装什么内容”

这样后面不管你是：

- 换主线
- 换测试剧本
- 换 UI 图片
- 加新事件
- 加新物品

都会更容易判断该去哪个目录动手。
