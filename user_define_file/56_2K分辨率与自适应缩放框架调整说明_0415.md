# 56_2K分辨率与自适应缩放框架调整说明_0415

## 1. 本次目标

本次改造按“先框架、后美术替换”的思路，先完成以下两件事：

1. 将项目 UI 设计基准从 `1920x1080` 提升到 `2560x1440`（2K）。
2. 在应用根层追加自适应画面缩放策略，支持常见横屏与竖屏窗口比例。

---

## 2. 本次改动总览

### 2.1 分辨率基线提升到 2K

- `project.godot` 的视口基准由 `1920x1080` 改为 `2560x1440`。
- `MainUiLayoutSettings` 与 `main_ui_layout.tres` 的 `WindowBaseWidth/WindowBaseHeight` 同步改为 `2560/1440`。

### 2.2 UI 尺寸参数按 2K比例放大

为了保持在 2K 下的视觉占比，本次将核心布局与字号参数按约 `4/3` 放大。

- 放大入口 1：`Resources/UI/main_ui_layout.tres`
- 放大入口 2：`Configs/UI/gameplay_ui_theme.yaml`

这样在 2K 下不会出现“布局突然显得过小”的问题；在较低分辨率下则通过缩放策略回落。

### 2.3 新增自适应缩放策略（AppRoot）

`Scripts/UI/AppRoot.cs` 的窗口缩放策略改为：

1. 统一使用 `CanvasItems` 缩放模式。
2. 统一以 `WindowBaseWidth/Height` 作为内容基准尺寸（现在是 2K）。
3. 常规横屏使用 `Expand`，优先利用不同宽高比下的额外可视区域。
4. 竖屏窗口（宽高比小于 1.2）切换为 `KeepWidth`，优先保证阅读宽度稳定。
5. 监听窗口 `SizeChanged`，在运行时动态重算缩放策略。

## 5. 自适应缩放规则（当前版本）

当前规则由 `AppRoot.UpdateAdaptiveScale()` 控制：

1. 获取当前窗口宽高比。
2. 当窗口接近竖屏（`aspect < 1.2`）时：
   - 使用 `KeepWidth`。
3. 其他常见横屏比例（16:9、16:10、超宽等）：
   - 使用 `Expand`。
4. `ContentScaleFactor` 固定为 `1.0`，避免额外倍数叠加。

## 7. 说明与后续建议

1. 本次属于“UI 框架层改造”，还未引入新的美术资源风格。
2. 后续接入 Stitch（或其他设计工具）导出的 UI 资源时，建议优先采用：
   - 可切片（9-slice）
   - 组件化导出
   - 文本与背景分离
   以降低非 16:9 设备上的拉伸风险。
3. 若后续需要进一步强化手机竖屏体验，可在下一轮补充“窄屏专用布局参数集”（例如独立的 mobile theme 配置）。
