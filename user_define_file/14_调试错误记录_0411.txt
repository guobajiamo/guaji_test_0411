《调试错误记录 0411》

这份文档记录的是：
在 2026-04-11 这次调试中，项目为什么会出现“灰色界面 / 按钮点击无反应 / 标签页无法顺利切换”的问题，以及我是如何定位并修复这些问题的。


================================
一、本次用户实际遇到的问题
================================

你这次反馈的现象主要有 3 类：

1. 游戏启动后只看到灰色界面，最初无法正常运行
2. 进入界面后，点击“捡起石斧”没有看到任何变化
   - 按钮没有消失
   - “点击砍树”“挂机砍树”没有出现
   - 背包里没有看到石斧
3. 中间区域的标签页点击切换不顺利，看起来像是没反应


================================
二、第一层根因：项目当时缺少 C# 工程文件
================================

最初灰色界面的根因，不是 UI 场景坏了，而是：

项目根目录当时缺少：

`test_00_0410.csproj`

这会导致 Godot 虽然能看到 `.cs` 脚本文件，
但无法把它们真正编译成项目程序集。

运行时表现就会变成：

1. `GameManager.cs` 自动加载失败
2. `SignalBus.cs` 自动加载失败
3. `MainUI.cs` 找不到对应类
4. 画面只剩下一个灰色界面或空界面


------------------------------
我是怎么确认这一点的
------------------------------

我做了两步检查：

1. 直接用 Godot 无界面构建
确认 Godot 可以找到项目，但项目程序集没有正常加载

2. 直接用 `dotnet build`
进一步发现项目里根本没有可以编译的 Godot C# 工程文件


------------------------------
我是怎么修复的
------------------------------

我新增了：

`test_00_0410.csproj`

里面指定了：
- `Godot.NET.Sdk/4.6.2`
- `net8.0`
- 动态加载
- 程序集名 `test_00_0410`

然后又顺手修复了编译里暴露出来的几个真实错误：

1. `Scripts/Core/Helpers/SortingHelper.cs`
- 补了 `using System;`

2. `Scripts/Core/Helpers/YamlConfigLoader.cs`
- 修正了 `Select(ConvertToString)` 的写法

3. `Scripts/UI/InventoryPanel.cs`
- 修正了 `GetValueOrDefault` 的使用方式

4. `Scripts/Systems/IdleSystem.cs`
- 补了空值保护，把编译警告清成 0

修复后结果：

`dotnet build`
已经达到：

- `0` 警告
- `0` 错误

并且 Godot 启动日志里已经能看到：

- `静态配置加载完成`
- `初版游戏已完成启动`


================================
三、第二层根因：UI 刷新方式过于激进，导致交互被打断
================================

当项目已经能正常启动后，
你又遇到了新的问题：

点击“捡起石斧”后，界面看起来没有反应；
中间标签页点击切换也不顺利。

这一次的关键点是：

“程序逻辑其实大体是通的，但 UI 层每隔 0.2 秒就在重建一大批控件，导致用户的鼠标交互体验很差。”


------------------------------
为什么我会这样判断
------------------------------

我先没有直接猜，而是做了一个临时自动化探针来验证：

1. 在 Godot 内部自动找到“捡起石斧”按钮
2. 直接程序化触发一次 `pressed`
3. 观察界面数据有没有变化

验证结果是：

1. `执行事件成功：捡起石斧`
   日志会正常出现

2. 背包摘要会变成：
   `已显示 1 种物品。`

3. “点击砍树”和“挂机砍树”按钮都会出现

4. `TabContainer.current_tab`
   可以被正常切换，且切换后能够保持

这说明：

1. 点击事件系统本身是通的
2. 背包刷新逻辑是通的
3. 标签页切换机制本身是通的

也就是说：

“不是逻辑没写，而是界面控件在刷新循环里被反复清空重建，导致真实鼠标点击时，用户很容易感觉到点了没反应、切不了页。”


------------------------------
最典型的问题点
------------------------------

问题集中在：

`Scripts/UI/MainUI.cs`

旧逻辑里：

`_Process()`
会每隔很短时间就调用：

`RefreshAllPanels()`

而 `RefreshAllPanels()` 又会继续调用：

1. `ButtonListPanel.RebuildButtons(...)`
2. `InventoryPanel.RefreshInventory()`
3. `SkillPanel.RefreshSkills()`
4. `DictionaryPanel.RefreshDictionary()`
5. `FactionPanel.RefreshFactions()`
6. `ZoneSelectPanel.RefreshZones()`
7. `LogPanel.SetMessages(...)`

这里面很多方法的旧写法，都是：

1. 先 `Clear`
2. 再删掉旧子节点
3. 再重新 `AddChild`

这会导致一个实际问题：

用户鼠标正在点某个控件时，
这个控件可能恰好被下一轮刷新删掉又重建，
所以就表现成：

- 按钮明明在眼前，但点击很不稳定
- 标签页明明存在，但切换体验很差


================================
四、这次我怎么修复 UI 交互问题
================================

修复思路不是“把刷新彻底关掉”，
而是改成：

“只有数据真的变了，才去更新；能复用旧控件，就不要删掉重建。”


------------------------------
1. 事件按钮改成优先复用旧按钮
------------------------------

修改文件：

1. `Scripts/UI/EventButtonItem.cs`
2. `Scripts/UI/ButtonListPanel.cs`

具体修复：

在 `EventButtonItem.cs` 中：
- 拆出了 `UpdateView(...)`
- 不再每次重建都重新绑定一套匿名点击回调
- 改成内部统一的 `OnPressedInternal()`

在 `ButtonListPanel.cs` 中：
- 先把按钮数据转成列表
- 如果现有按钮和新数据的 `EventId` 序列没变
  -> 就直接更新按钮文本、提示、禁用状态
  -> 不再清空整个列表
- 只有按钮结构真的变化时，才清空重建
- 如果本来就在显示“当前没有可用按钮”，也不再每 0.2 秒重建一次空状态标签

修复效果：

左侧按钮不再在刷新循环里被反复销毁重建，
鼠标点击稳定性大幅提升。


------------------------------
2. 背包、技能、图鉴、势力、区域改成“有变化才重绘”
------------------------------

修改文件：

1. `Scripts/UI/InventoryPanel.cs`
2. `Scripts/UI/SkillPanel.cs`
3. `Scripts/UI/DictionaryPanel.cs`
4. `Scripts/UI/FactionPanel.cs`
5. `Scripts/UI/ZoneSelectPanel.cs`

具体修复：

每个面板都新增了自己的“签名字符串”缓存，例如：

- 背包：按 `物品ID + 数量 + 是否获得` 生成签名
- 技能：按 `技能ID + 等级 + 当前经验 + 是否可升级` 生成签名
- 图鉴：按已解锁物品 ID 列表生成签名
- 势力：按声望和和平状态生成签名
- 区域：按解锁、探索度、通关次数生成签名

刷新时先比较：

1. 如果签名没变
   -> 直接 return
2. 如果签名变了
   -> 才真正清空并重建内容

修复效果：

中间区域不会再因为“其实数据没变”而反复删节点、重建节点，
所以标签页交互也不会被无意义刷新频繁干扰。


------------------------------
3. 日志面板也改成“文本没变就不刷新”
------------------------------

修改文件：

`Scripts/UI/LogPanel.cs`

修复内容：

- 新增 `_lastMessageSignature`
- 如果整段日志文本和上次一样，就不重复写入 `TextEdit`

修复效果：

日志区域不再频繁强制改写内容和光标位置，
减少了 UI 抖动。

七、最后的结论
================================

这次问题的本质可以分成两层：

第一层：
项目一开始连 C# 工程文件都缺失，所以根本没法正常启动。

第二层：
启动修好后，游戏逻辑其实已经能跑，但 UI 刷新方式过于激进，导致按钮和面板交互体验异常，表现得像“点了没反应”。

修复完成后，当前项目已经达到：

1. 能正常编译
2. 能正常启动
3. “捡起石斧”这条事件链已确认能触发
4. “点击砍树”“挂机砍树”按钮会在首个事件后出现
5. 标签页切换机制已确认可用，并且不会在刷新中立刻被重置

如果后续还要继续调试“挂机中资源实时刷新感受”或“商店 UI / 更细的标签页体验”，建议继续沿用本次这个原则：

“优先复用已有控件，避免在高频刷新里整棵重建 UI。”

八、追加记录：背包物品显示与经验显示问题（同日后续调试）

本段记录的是：
在前面问题修复后，你继续测试时发现的新问题，以及这次我是如何继续定位和修复的。


------------------------------
1. 你反馈的新现象
------------------------------

这次你反馈的现象有 3 个：

1. 点击“获取石斧”后，
   背包栏下方文字会变成“已显示1种物品”，
   但实际看不到任何物品栏

2. 点击“挂机砍树”后，
   每次读条从 0 到 100% 的瞬间，
   背包里会短暂出现“石斧”和“原木 x 数量”，
   但很快又被刷掉消失

3. 当砍树经验攒够 180、点了“手动升级”后，
   技能经验显示突然从整数变成浮点数，
   视觉上会显得比较奇怪


------------------------------
2. 背包物品“短暂出现又消失”的真正根因
------------------------------

这次问题的根因非常明确，出在：

`Scripts/UI/InventoryPanel.cs`

我在检查代码时发现，
`RefreshInventory()` 里有这样一条逻辑问题：

1. 方法一开始先执行了：
   `ClearItemList()`

2. 然后才去计算当前背包签名 `nextSignature`

3. 如果签名没有变化，就直接 `return`

这就会造成一个非常典型的问题：

1. 上一次刷新刚把物品列表显示出来
2. 下一次刷新一开始先把列表清空
3. 因为这一次背包数据其实没变，所以签名也没变
4. 结果代码直接 `return`
5. 界面就停留在“列表已经被清空，但没有重建”的状态

这正好和你看到的现象完全一致：

- 文字说“已显示 1 种物品”
- 但列表里没有物品
- 当挂机结算让数量变化时，签名变化，列表会短暂重建出来
- 紧接着下一次刷新又被清空并提前返回，所以又消失


------------------------------
3. 我是怎么修复背包显示问题的
------------------------------

修改文件：

1. `Scripts/UI/InventoryPanel.cs`
2. `Scripts/UI/ItemSlotUI.cs`

修复内容：

在 `InventoryPanel.cs` 中：

1. 删除了 `RefreshInventory()` 开头那次错误的 `ClearItemList()`
2. 改成先计算签名，再判断是否真的需要重绘
3. 如果签名没变：
   - 保留现有物品列表
   - 不再错误清空
4. 把摘要文字：
   `已显示 X 种物品`
   放到签名判断之前更新
   这样即使列表内容不需要重绘，摘要文字也能保持正确

在 `ItemSlotUI.cs` 中：

1. 给物品按钮补了：
   - `SizeFlagsHorizontal = ExpandFill`
   - `CustomMinimumSize = new Vector2(0, 40)`
   - `Alignment = HorizontalAlignment.Left`

这样可以让背包物品栏的可见性更稳定，
避免因为默认尺寸过小导致视觉上不好辨认。


------------------------------
4. 技能经验变成浮点数的根因
------------------------------

这次经验显示变成浮点数，
不是 UI 自己乱了，而是前面的挂机经验结算逻辑本来就用了小数。

问题出在：

`Scripts/Systems/IdleSystem.cs`

旧逻辑里：

`totalExp = completedCycles * interval`

而技能表 `skills.yaml` 里的 `interval`：
- 1 级是 `6.000`
- 2 级开始就变成了 `5.789`、`5.579` 这种小数

这意味着：

1. 1 级时你看到的经验增长刚好还是整数
2. 一旦升级到 2 级，
   每轮经验就会开始按 `5.789` 这种小数累加
3. 所以 UI 自然会开始显示浮点数

从“程序逻辑上”这不是崩溃错误，
但从“文字挂机游戏的直观体验”上，
确实会让人感觉很奇怪。


------------------------------
5. 我是怎么修复经验显示问题的
------------------------------

修改文件：

1. `Scripts/Systems/IdleSystem.cs`
2. `Scripts/UI/SkillPanel.cs`
3. `Scripts/UI/CharacterStatusPanel.cs`

修复内容：

在 `IdleSystem.cs` 中：

我把每轮经验改成了“按间隔时间四舍五入后的整数值”：

1. 先算：
   `expPerCycle = Round(levelEntry.Interval)`
2. 再算：
   `totalExp = completedCycles * expPerCycle`

同时保底至少为 `1`

这样做的效果是：

1. 首版的经验增长重新回到整数节奏
2. 升级阈值本来就是整数，更符合当前系统的直觉
3. 以后如果真要做更复杂的小数经验体系，再重新设计也更清楚

另外我还顺手把显示层也做了收口：

在 `SkillPanel.cs` 和 `CharacterStatusPanel.cs` 中：

1. 新增了 `FormatExpValue(...)`
2. 如果经验值本质上已经接近整数，就按整数显示
3. 只有确实存在小数时，才显示 `0.###` 形式

这样即使以后某些分支又引入了小数经验，
界面显示也不会显得太突兀。
