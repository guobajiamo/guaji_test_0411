# 31_新UI版本_剧本与区域场景配置方法说明文档

## 1. 先说最重要的结论

当前项目已经不只是“剧本内容可以 YAML 化”，而是：

- 剧本内容本身可以 YAML 化
- 新 UI 下“多级区域 / 二级区域 / 子场景 / 事件显示归属”也可以 YAML 化

你后面如果要扩写新剧本或给现有剧本补内容，最常见的工作会集中在：

1. 新建或复制一个剧本 `.tres`
2. 给它准备 `Configs/Scenarios/你的剧本名/` 的内容配置目录
3. 在 `Items / Skills / Events / Factions / Zones / Localization` 里写玩法内容
4. 在 `UI/gameplay_layout.yaml` 里写多级区域、二级区域、子场景，以及每个子场景里显示哪些事件
5. 如有需要，在 `UI/info_panel.yaml` 里写右上角说明内容

也就是说，当前推荐的工作方式是：

- 系统逻辑尽量少改
- 剧本内容优先改 YAML
- 新 UI 布局优先改 `gameplay_layout.yaml`

## 2. 当前真正应该改的剧本配置目录在哪里

### 2.1 主线剧本

主线剧本资源文件：

- `Resources/Scenarios/main_story_scenario.tres`

主线剧本当前使用的 YAML 根目录：

- `Configs/Scenarios/MainStory/`

### 2.2 测试剧本

测试剧本资源文件：

- `Resources/Scenarios/test_game_scenario.tres`

测试剧本当前使用的 YAML 根目录：

- `Configs/Scenarios/TestGame/`

## 3. 新建一个新剧本，建议按这个步骤做

### 第一步：复制一个现有剧本资源

最稳妥的做法，是先复制：

- `Resources/Scenarios/main_story_scenario.tres`

例如你想做一个新剧本 `forest_story`，可以新建：

- `Resources/Scenarios/forest_story_scenario.tres`

### 第二步：修改剧本资源里的基础字段

你需要重点改这些字段：

- `ScenarioId`
- `DisplayName`
- `Description`
- `ShowInNewGameMenu`
- `IsDefaultStoryScenario`
- `IsTestScenario`
- `EnableSystemTab`

示例：

```text
ScenarioId = "forest_story"
DisplayName = "森林支线剧本"
Description = "用于测试森林区域扩写的版本。"
ShowInNewGameMenu = true
IsDefaultStoryScenario = false
IsTestScenario = false
EnableSystemTab = true
```

### 第三步：给新剧本准备独立目录

建议新建：

```text
Configs/Scenarios/ForestStory/
  Items/
    categories.yaml
    items.yaml
  Skills/
    skills.yaml
  Events/
    oneshot_events.yaml
    click_events.yaml
    idle_events.yaml
  Factions/
    factions.yaml
  Zones/
    zones.yaml
  Localization/
    locale_zh.yaml
  UI/
    info_panel.yaml
    gameplay_layout.yaml
```

### 第四步：把 `.tres` 指向你的 YAML

剧本资源里需要指向：

- `CategoriesConfigPath`
- `ItemsConfigPath`
- `SkillsConfigPath`
- `OneshotEventsConfigPath`
- `ClickEventsConfigPath`
- `IdleEventsConfigPath`
- `FactionsConfigPath`
- `ZonesConfigPath`
- `LocalizationPath`
- `InfoPanelConfigPath`

注意：

- 当前并没有单独的 `GameplayLayoutPath`
- 程序会根据 `InfoPanelConfigPath` 所在目录
- 自动去找同目录下的 `gameplay_layout.yaml`

所以只要你把：

- `InfoPanelConfigPath = "res://Configs/Scenarios/ForestStory/UI/info_panel.yaml"`

写对了，那么：

- `res://Configs/Scenarios/ForestStory/UI/gameplay_layout.yaml`

就会被自动读取。

## 4. 如何配置物品、技能、货币、经验值

这部分核心逻辑与旧版文档基本一致，仍然按下面方式配置。

### 4.1 物品分类 `categories.yaml`

示例：

```yaml
categories:
  - id: "tool"
    parent_id: ""
    name_key: "工具"
    description_key: "用于采集与互动的工具"
    definition_order: 100
    is_visible: true
```

### 4.2 物品 `items.yaml`

示例：

```yaml
items:
  - id: "tool_stone_axe"
    parent_id: "tool"
    name_key: "石斧"
    description_key: "最基础的伐木工具"
    definition_order: 100
    base_rarity: "Common"
    base_max_stack: 1
    sell_price: 0
    tags: ["Tool", "Axe"]
    tool_bonuses:
      log_yield_multiplier: 1.0
      chop_speed_multiplier: 1.0
```

### 4.3 技能 `skills.yaml`

示例：

```yaml
skills:
  - id: "skill_chopping"
    name_key: "砍树"
    description_key: "伐木技能"
    max_level: 20
    initial_level: 1
    max_total_exp: 34200
    required_tool_tag: "Axe"
    primary_output_item_id: "mat_wood_log"
    level_table:
      - { level: 1, exp_to_next: 180, output: 5.000, interval: 6.000 }
```

### 4.4 货币

当前正式跑通的主货币仍然是：

- 金币 / Gold

你可以通过 YAML 配：

- 条件 `HasGold`
- 效果 `GrantGold / RemoveGold`
- 商店 `payment_type: Gold`
- 商店 `gold_cost`

`ExtraCurrencies` 仍然只是预留状态，还没有完整做成独立货币表。

### 4.5 经验值

技能经验目前仍然主要来自：

- 事件效果 `GrantSkillExp`
- 挂机事件结算

技能升级经验仍由：

- `skills.yaml -> level_table.exp_to_next`

控制。

## 5. 如何配置事件

事件文件仍然分成三类：

- `oneshot_events.yaml`
- `click_events.yaml`
- `idle_events.yaml`

### 5.1 一次性事件示例

```yaml
events:
  - id: "evt_pickup_stone_axe"
    name_key: "捡起石斧"
    description_key: "你发现了一把石斧"
    hover_info_key: "获得石斧"
    type: "OneshotClick"
    prerequisites: []
    costs: []
    effects:
      - { effect_type: "GrantItem", target_id: "tool_stone_axe", int_value: 1 }
    button_list_group: "MainOneshot"
```

### 5.2 单次点击事件示例

```yaml
events:
  - id: "evt_click_chop"
    name_key: "点击砍树"
    description_key: "手动砍一下树"
    hover_info_key: "获得原木"
    type: "RepeatableClick"
    prerequisites:
      - { condition_type: "HasItem", target_id: "tool_stone_axe", required_value: 1 }
    costs: []
    effects:
      - { effect_type: "GrantItem", target_id: "mat_wood_log", int_value: 1 }
    button_list_group: "MainClick"
```

### 5.3 挂机事件示例

```yaml
events:
  - id: "evt_idle_chop"
    name_key: "挂机砍树"
    description_key: "自动持续砍树"
    hover_info_key: "持续获得原木与砍树经验"
    type: "IdleLoop"
    prerequisites:
      - { condition_type: "HasItem", target_id: "tool_stone_axe", required_value: 1 }
    linked_skill_id: "skill_chopping"
    button_list_group: "MainIdle"
```

### 5.4 要特别说明的新变化

在旧 UI 里，`button_list_group` 更像是“显示在哪一列按钮里”。

但在当前新 UI 里，真正决定事件显示位置的已经不是它，而是：

- `UI/gameplay_layout.yaml` 里的 `scene.event_ids`

也就是说：

- `button_list_group` 现在更偏向保留语义兼容
- 实际显示在哪个子场景中，要看 `gameplay_layout.yaml`

当前新 UI 下按钮的排序与颜色，主要看：

- `EventType`

对应关系是：

- `OneshotClick`：红色、排序最高
- `RepeatableClick`：绿色、排序中间
- `IdleLoop`：橙色、排序最低

## 6. 如何在新 UI 下创建多级区域

这是这轮最关键的新内容。

当前多级区域配置文件路径是：

- `Configs/Scenarios/对应剧本/UI/gameplay_layout.yaml`

### 6.1 顶层结构

它的基本结构是：

```yaml
default_area_id: "market_street"

regions:
  - id: "human_village"
    title: "人间之里"
    accent: "primary_story"
    expanded_by_default: true
    areas:
      - id: "market_street"
        title: "商店街"
        accent: "secondary_peace"
        scenes:
          - id: "dessert_house"
            title: "琉璃点心屋"
            description: "人间之里里最热闹的小店之一。"
            event_ids:
              - "evt_read_chopping_notice"
              - "evt_stranger_reward_offer"
```

### 6.2 一级地区怎么配

一级地区就是 `regions` 下的一项。

示例：

```yaml
- id: "human_village"
  title: "人间之里"
  accent: "primary_story"
  expanded_by_default: true
  areas: []
```

字段含义：

- `id`：一级地区唯一 ID
- `title`：左侧显示名称
- `accent`：文字颜色类型
- `expanded_by_default`：默认是否展开
- `areas`：这个一级地区下有哪些二级区域

当前常用的 `accent` 取值有：

- `primary_story`
- `primary_faction`
- `secondary_peace`
- `secondary_battle`

这些颜色来自：

- `Configs/UI/gameplay_ui_theme.yaml -> region_accents`

### 6.3 二级区域怎么配

二级区域写在一级地区的 `areas` 里。

示例：

```yaml
areas:
  - id: "market_street"
    title: "商店街"
    accent: "secondary_peace"
    scenes: []
```

字段含义：

- `id`：二级区域唯一 ID
- `title`：左侧显示名称
- `accent`：区域文字颜色
- `scenes`：这个二级区域下有哪些子场景

如果你想新建一个二级区域，例如“稗田邸”，可以写成：

```yaml
areas:
  - id: "hida_house"
    title: "稗田邸"
    accent: "secondary_peace"
    scenes:
      - id: "archive_room"
        title: "书库"
        description: "用于摆放图鉴、委托、剧情相关内容。"
        event_ids: []
```

### 6.4 `default_area_id` 的作用

`default_area_id` 决定：

- 玩家进入这个剧本时，默认激活哪个二级区域

例如：

```yaml
default_area_id: "market_street"
```

表示：

- 刚进入剧本时
- 左侧默认选中“商店街”
- 中部也会立即切换到“当前区域”页并显示“商店街”下的子场景

## 7. 如何给每个二级区域配置对应的子场景

这也是你后面最常用的配置点。

### 7.1 子场景的基本写法

子场景写在二级区域的 `scenes` 列表里。

示例：

```yaml
scenes:
  - id: "dessert_house"
    title: "琉璃点心屋"
    description: "人间之里里最热闹的小店之一。"
    event_ids:
      - "evt_read_chopping_notice"
      - "evt_stranger_reward_offer"
```

字段含义：

- `id`：子场景唯一 ID
- `title`：中部卡片标题
- `description`：场景说明
- `event_ids`：这个场景里要显示哪些事件按钮

### 7.2 以“琉璃点心屋”为例怎么理解

例如你有一个二级区域“商店街”，它下面要挂“琉璃点心屋”“甘味茶屋”“孙惣刃物锻冶”三个子场景，你可以这样写：

```yaml
areas:
  - id: "market_street"
    title: "商店街"
    accent: "secondary_peace"
    scenes:
      - id: "dessert_house"
        title: "琉璃点心屋"
        description: "人间之里里最热闹的小店之一。"
        event_ids:
          - "evt_read_chopping_notice"
          - "evt_stranger_reward_offer"

      - id: "tea_house"
        title: "甘味茶屋"
        description: "适合放慢节奏的休息点。"
        event_ids:
          - "evt_pickup_stone_axe"

      - id: "forge_shop"
        title: "孙惣刃物锻冶"
        description: "这里暂时还没有可互动内容。"
        event_ids: []
```

运行时效果是：

- 左侧点中“商店街”
- 中部“当前区域”页会出现三张卡片
- 每张卡片就是一个子场景
- 每张卡片里显示自己 `event_ids` 指向的事件按钮

### 7.3 子场景与事件之间的关系

当前新 UI 下：

- 事件定义文件里只负责定义“事件本身是什么”
- `gameplay_layout.yaml` 负责定义“事件显示在哪个子场景里”

这意味着：

- 同一个事件 ID 不要重复挂到多个场景里，除非你就是想让它重复出现在多个地方
- 你以后要调整显示位置，不需要改事件 YAML，只需要改 `event_ids`

## 8. 新 UI 下一次性事件弹窗是否仍然适用

结论先说：

- 18 号文档里的“一次性事件弹出对话框两种情况的配置方案”
- 在现在这套新 UI 逻辑下仍然适用

### 8.1 为什么仍然适用

因为现在的弹窗逻辑仍然是：

- `GamePlayUI.OnEventButtonPressed(eventId)`
  -> `ShowDialogForEvent(...)`
  -> `EventDialogPanel`

也就是说：

- 事件弹窗的底层触发方式并没有因为新 UI 改版而变化
- 只是事件按钮从“旧版左侧三列事件区”
- 改成了“当前区域页里某个子场景卡片中的按钮”

### 8.2 18 号文档里需要更新理解的地方

18 号文档里的思路仍然成立，但要补两点新理解：

#### 第一，配置路径要看剧本目录

旧文档里写的是：

- `Configs/Events/oneshot_events.yaml`

而在当前版本里，你真正应该改的是：

- `Configs/Scenarios/MainStory/Events/oneshot_events.yaml`
- 或 `Configs/Scenarios/TestGame/Events/oneshot_events.yaml`

#### 第二，事件显示位置要再配到子场景

旧文档里更偏向默认事件会出现在主界面按钮区。

而在当前版本里，如果你只在 `oneshot_events.yaml` 里写了事件，但没有把它挂到：

- `UI/gameplay_layout.yaml -> 某个 scene -> event_ids`

那么这个事件不会自动显示出来。

所以在新 UI 下，一次性事件要“能看见、能点到”，通常需要两步都完成：

1. 在 `Events/oneshot_events.yaml` 里定义事件本身
2. 在 `UI/gameplay_layout.yaml` 里把这个事件 ID 填进某个子场景的 `event_ids`

### 8.3 单确认弹窗模式仍然怎么配

示例：

```yaml
dialog:
  body_text: "你终于拿到了第一把石斧。"
  confirm_button_text: "我知道了"
```

这类弹窗现在仍会自动带一个：

- `我再想想`

玩家点这个按钮会关闭弹窗，不消耗一次性事件。

### 8.4 双选项分支弹窗模式仍然怎么配

示例：

```yaml
dialog:
  body_text: "路人想送你谢礼。"
  consume_source_event_on_choice: true
  choices:
    - { button_text: "接受馈赠", target_event_id: "evt_accept_gift_gold" }
    - { button_text: "婉拒谢礼", target_event_id: "evt_decline_reward_food" }
```

这种模式仍然推荐：

- 1 个源弹窗事件
- 2 个隐藏目标事件

目标事件通常建议：

- `button_list_group: "None"`

并且不要把目标事件再挂到 `gameplay_layout.yaml` 里，否则它们会直接显示成普通按钮。

## 9. 当前最适合你的配置工作流

如果你继续扩写新剧本，推荐按这个顺序做：

1. 先复制一个剧本 `.tres`
2. 先复制一整套 `Configs/Scenarios/...` 子目录
3. 先填 `Localization/locale_zh.yaml`
4. 再补 `categories.yaml` 和 `items.yaml`
5. 再补 `skills.yaml`
6. 再补 `oneshot/click/idle_events.yaml`
7. 再补 `factions.yaml` 和 `zones.yaml`
8. 最后补 `UI/info_panel.yaml` 和 `UI/gameplay_layout.yaml`

这样最不容易乱。

## 10. 给你的一个非常实用的建议

因为你现在是“主线剧本”和“测试剧本”并行开发，我仍然建议始终遵守这条：

- 新内容先在 `Configs/Scenarios/TestGame/` 里试
- 确认玩法没问题后，再人工搬到 `Configs/Scenarios/MainStory/`

在新 UI 版本里，这条建议还要多补一层：

- 场景布局、区域层级、事件归属，也先在 `Configs/Scenarios/TestGame/UI/gameplay_layout.yaml` 里试

这样能最大限度避免：

- 测试事件误混进正式主线
- 主线区域布局还没设计好时被测试结构污染

## 11. 最后用一句话概括当前配置思路

现在最理想的理解方式是：

- `.tres` 负责“选哪套剧本”
- 内容 YAML 负责“这套剧本里有什么”
- `gameplay_layout.yaml` 负责“这些内容在新 UI 里摆到哪里”
- 系统脚本负责“这些内容按什么规则运行”

你后面如果只是：

- 扩剧情
- 加区域
- 加子场景
- 给某个场景挂事件
- 加一次性弹窗事件

优先改 YAML 就对了。
