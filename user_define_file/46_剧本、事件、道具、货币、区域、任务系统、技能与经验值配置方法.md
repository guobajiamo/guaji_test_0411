# 46_剧本、事件、道具、货币、区域、任务系统、技能与经验值配置方法

## 1. 这一版最重要的变化

当前项目除了原有的：

- 剧本 `.tres`
- 物品 / 技能 / 事件 / 势力 / 区域 / 本地化 YAML
- `UI/gameplay_layout.yaml`

现在又新增了两类正式配置入口：

1. `UI/Tabs/`
2. `Quests/Definitions/`

你可以把这一版理解成：

- `.tres` 决定“载入哪套剧本”
- 各类内容 YAML 决定“剧本里有什么”
- `UI/gameplay_layout.yaml` 决定“事件按钮摆在哪个子场景里”
- `UI/Tabs/` 决定“中间标签页显示什么、哪些页签暂时隐藏”
- `Quests/Definitions/` 决定“任务链怎么解锁、怎么推进、完成后能领什么奖励”

## 2. 新增的剧本资源字段

当前剧本资源除了原有路径，还新增了：

- `UiTabsConfigPath`
- `QuestsConfigPath`

例如测试剧本现在会指向：

```text
UiTabsConfigPath = "res://Configs/Scenarios/TestGame/UI/Tabs"
QuestsConfigPath = "res://Configs/Scenarios/TestGame/Quests/Definitions"
```

主线剧本同理。

## 5. 中间标签页现在怎么配

当前中间区域显示顺序已经固定为：

1. 当前场景
2. 背包
3. 技能
4. 战斗
5. 任务
6. 教学
7. 成就
8. 系统

图鉴接口仍保留，但当前默认不显示。

### 5.1 页签配置目录

每个剧本都可以在：

- `Configs/Scenarios/对应剧本/UI/Tabs/`

下拆成多个 YAML。

程序会把这个目录下的所有 `.yaml` 合并读取。

### 5.2 页签 YAML 示例

```yaml
tabs:
  - id: "quest"
    title: "任务"
    visible: true
    placeholder_title: "任务总览"
    content_lines:
      - "左列红色按钮显示主线任务。"
      - "右列绿色按钮显示支线任务。"
```

字段说明：

- `id`：页签 ID，当前支持：
  - `current_region`
  - `inventory`
  - `skills`
  - `battle`
  - `quest`
  - `tutorial`
  - `achievement`
  - `dictionary`
  - `system`
- `title`：页签标题
- `visible`：是否显示页签
- `placeholder_title`：占位页的大标题
- `content_lines`：占位说明文本

### 5.3 图鉴为什么不显示

图鉴页现在建议这样配置：

```yaml
tabs:
  - id: "dictionary"
    title: "图鉴"
    visible: false
```

这表示：

- 接口还在
- 代码不会删除
- 当前版本不显示图鉴页签
- 以后只要把 `visible` 改回 `true`，就能继续启用

## 6. `UI/gameplay_layout.yaml` 现在仍然只负责场景摆放

这一点没有变。

它仍然只负责：

- 一级区域
- 二级区域
- 子场景
- 每个子场景里挂哪些事件 ID

也就是说：

- 任务不是在 `gameplay_layout.yaml` 里定义
- 任务系统单独放到 `Quests/Definitions/`

## 7. 任务系统 YAML 放在哪里

任务定义统一放在：

- `Configs/Scenarios/对应剧本/Quests/Definitions/`

这个目录支持多文件拆分。

程序会自动把同目录下所有 `.yaml` 合并读取。

所以后面如果任务越来越多，推荐按：

- 主线一章一个文件
- 支线一组一个文件

去拆。

## 8. 任务 YAML 基础结构

示例：

```yaml
quests:
  - id: "quest_main_woodcutter_intro"
    category: "Main"
    chain_id: "main_story"
    chain_order: 10
    title: "初入人间之里"
    description: "先拿到石斧，读完伐木心得，再处理路人的谢礼。"
    unlock_condition_count: 0
    unlock_conditions: []
    required_completed_quest_ids: []
    step_count: 3
    steps:
      - id: "pickup_axe"
        event_id: "evt_pickup_stone_axe"
        title: "捡起石斧"
        description: "去人间之里的木匠屋拿到第一把石斧。"
    rewards:
      - { effect_type: "GrantGold", target_id: "", int_value: 20 }
      - { effect_type: "UnlockAchievement", target_id: "ach_testgame_story_intro", int_value: 1 }
```

## 9. 任务字段说明

### 9.1 顶层字段

- `id`：任务唯一 ID
- `category`：`Main` 或 `Side`
- `chain_id`：任务链分组 ID
- `chain_order`：同链中的顺序
- `title`：任务标题
- `description`：任务主要目的说明

### 9.2 解锁字段

- `unlock_condition_count`
- `unlock_conditions`
- `required_completed_quest_ids`

当前逻辑是：

- 同一 `chain_id` 下，后面的任务会等待前面的任务完成
- 如果额外写了 `required_completed_quest_ids`，也必须全部满足
- `unlock_conditions` 也必须全部满足
- 一旦任务解锁，解锁状态会写入存档

### 9.3 步骤字段

- `step_count`
- `steps`

每个 `step` 当前最重要的是：

- `event_id`

任务推进会按 `steps` 的顺序检查。

当前任务进度默认会卡在：

- 第一个尚未完成的步骤

已经连续完成的步骤会作为“已完成关键事件链”保存给任务界面查看。

### 9.4 奖励字段

- `rewards`

这里使用和事件 `effects` 一样的结构：

- `effect_type`
- `target_id`
- `int_value`
- `double_value`
- `text_value`

## 10. 当前任务奖励支持哪些 `effect_type`

当前任务奖励已经支持：

- `GrantItem`
- `RemoveItem`
- `GrantGold`
- `RemoveGold`
- `GrantSkillExp`
- `AddFactionReputation`
- `UnlockZone`
- `CompleteEvent`
- `CompleteQuest`
- `StartBattle`
- `UnlockAchievement`

其中这几个最常用：

### 10.1 发放道具

```yaml
- { effect_type: "GrantItem", target_id: "tool_fishing_rod", int_value: 1 }
```

### 10.2 发放金币

```yaml
- { effect_type: "GrantGold", target_id: "", int_value: 30 }
```

### 10.3 解锁成就占位

```yaml
- { effect_type: "UnlockAchievement", target_id: "ach_story_intro", int_value: 1 }
```

### 10.4 自动完成另一条任务

```yaml
- { effect_type: "CompleteQuest", target_id: "quest_side_helper", int_value: 1 }
```

## 11. 当前任务支持哪些解锁条件

任务解锁条件复用通用条件判定器，所以当前支持：

- `None`
- `HasItem`
- `HasGold`
- `SkillLevel`
- `FactionReputation`
- `ZoneCleared`
- `EventCompleted`
- `QuestCompleted`
- `ItemAcquired`
- `BattleStatReached`

### 11.1 `ItemAcquired` 和 `HasItem` 的区别

这是这次任务系统里非常关键的一点。

`HasItem` 表示：

- 背包里此刻还持有这个道具

`ItemAcquired` 表示：

- 玩家历史上曾经拿到过这个道具一次

所以如果你要表达：

- “第一次拿到鱼竿后，解锁后续任务”

更推荐写：

```yaml
unlock_conditions:
  - { condition_type: "ItemAcquired", target_id: "tool_fishing_rod", required_value: 1 }
```

这样即使玩家后来把鱼竿卖掉或消耗掉，任务也不会重新锁回去。

### 11.2 `BattleStatReached`

这是这次专门预留给战斗系统的任务接口。

示例：

```yaml
unlock_conditions:
  - { condition_type: "BattleStatReached", target_id: "attack_power", required_value: 20 }
```

当前说明：

- 接口已经在运行态里预留
- 后续战斗系统只需要把对应战斗属性写入玩家档
- 任务系统就能直接读取并解锁任务

## 12. 任务界面当前的真实行为

当前任务页已经实现：

- 左列红色按钮显示主线任务
- 右列绿色按钮显示支线任务
- 鼠标悬浮任务按钮时，右上角信息面板同步显示任务说明
- 详情区显示当前任务描述与当前进度
- 按住 `Shift` 可展开查看已完成关键事件链
- 已完成且有奖励的任务，可以在任务页手动领取奖励

另外顶部左上角原本的：

- `当前剧本：XX`

现在已经改成：

- `当前主线任务：XXXX`

## 13. 哪些任务状态会进存档

这次任务系统已经把这些状态纳入存档：

- 任务是否已解锁
- 任务是否已完成
- 任务奖励是否已领取
- 已完成任务 ID
- 已解锁成就 ID 占位
- 战斗属性占位字典

所以读档后不会丢失：

- 已经解锁但暂时没打开任务页的任务
- 已完成但还没领奖的任务
- 已经领过奖励的任务

## 14. 当前页签与任务系统最适合的维护方式

推荐工作流：

1. 先在 `Events/` 里把关键一次性事件写好
2. 再在 `UI/gameplay_layout.yaml` 里把这些事件挂到对应子场景
3. 再去 `Quests/Definitions/` 里把这些事件串成任务链
4. 如果要改标签显示，再去 `UI/Tabs/` 调整

这样最不容易混乱。

## 15. 关于货币与经验值，这一版怎么理解

### 15.1 货币

当前正式跑通的主货币仍然是：

- 金币

任务奖励也可以直接发金币：

```yaml
- { effect_type: "GrantGold", target_id: "", int_value: 50 }
```

### 15.2 技能经验

任务奖励也可以直接发技能经验：

```yaml
- { effect_type: "GrantSkillExp", target_id: "skill_fishing", int_value: 20 }
```

技能本体成长规则仍然继续写在：

- `Skills/Definitions/`

## 16. 当前最值得记住的一句话

这一版之后，新增内容时最常见的改法已经变成：

- 事件按钮去哪儿显示，改 `UI/gameplay_layout.yaml`
- 中间页签显不显示，改 `UI/Tabs/`
- 某几个关键事件怎么串成任务，改 `Quests/Definitions/`
- 任务完成后发什么奖励，继续改 `Quests/Definitions/`

也就是说：

- UI 摆放
- 页签显示
- 任务链逻辑

现在都已经有了独立 YAML 入口，不需要再优先去改系统脚本。
