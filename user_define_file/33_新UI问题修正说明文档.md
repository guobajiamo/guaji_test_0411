## 2. 每个问题为什么会出现，以及这次怎么修

### 2.1 左侧区域列表没有悬浮说明

#### 原因

之前左侧区域树里的一级区域按钮、二级区域按钮，只做了点击和展开/收起逻辑，没有给这些控件配置：

- `TooltipText`
- 鼠标进入/离开时的右上角信息联动

所以鼠标移上去时不会出现悬浮框，也不会把说明同步到右上角信息区。

#### 解决方式

这次在 `Scripts/UI/GamePlayUI.Layout.cs` 里给左侧区域树新增了悬浮说明逻辑：

- 一级区域：显示它包含哪些二级区域
- 二级区域：显示它包含多少子场景、当前可互动内容数量，以及子场景名称

并且这些悬浮说明现在会：

- 直接作为 Godot 的 tooltip 弹出
- 同步写到右上角信息区

## 2.2 点击过按钮后，其他按钮经常要点很多次才响应

#### 原因

这是这次最核心的 bug。

之前 `GamePlayUI` 在 `_Process` 里每隔 `0.2` 秒就执行一次整页刷新，而整页刷新会重新构建很多交互控件，包括：

- 左侧区域树按钮
- 中部当前区域页的场景卡片
- 场景里的事件按钮

这样一来，如果玩家刚好在鼠标按下到抬起之间遇到一次刷新：

- 原按钮会被销毁
- 新按钮会被重新创建
- Godot 的 `Pressed` 信号就可能丢失

玩家的主观感受就会变成：

- 明明点到了
- 但按钮像是“经常不响应”
- 要点很多次才成功一次

#### 解决方式

这次在 `Scripts/UI/GamePlayUI.cs` 里做了两层修正：

1. 把定时刷新改成 `RefreshTimedPanels()`，避免每次都做完全相同的大刷新流程。
2. 新增 `IsInteractiveRefreshDeferred()`，只要当前有鼠标按键处于按下状态，就暂时跳过这次定时刷新。

这样就避免了“玩家正在点击时，按钮被后台刷新重建掉”的问题。

这也是为什么这轮修正之后，按钮点击响应会明显稳定。

## 2.3 为什么区域层级会写在 `gameplay_layout.yaml`，而不是全放进 `zones.yaml`

#### 原因

这个问题本质上不是“写错文件”，而是“两个文件名里都有区域概念，但职责不同”，所以很容易混淆。

当前项目里：

- `Zones/zones.yaml` 负责玩法系统层的区域定义
- `UI/gameplay_layout.yaml` 负责新 UI 布局层的区域树结构

如果没有把这件事讲清楚，就会让人误以为“一级区域人间之里、二级区域商店街”这类内容全都应该进 `zones.yaml`。

#### 正确理解

`zones.yaml` 负责的是：

- 区域 ID
- 区域说明
- 最大通关次数
- 区域解锁链
- 解锁条件

也就是说，它偏向：

- 探索
- 战斗
- 清关
- 解锁规则

而 `gameplay_layout.yaml` 负责的是：

- 左侧区域树怎么长
- 一级区域下有哪些二级区域
- 二级区域下有哪些子场景
- 某个事件按钮应该出现在哪个子场景里

也就是说，它偏向：

- 内容界面布局
- 场景组织方式
- 按钮归属位置

#### 这次的处理结论

这次我没有把这些“一级区域 / 二级区域 / 子场景”移回 `zones.yaml`，因为它们在当前新 UI 架构里写在 `gameplay_layout.yaml` 是正确的。

但是我已经在修订后的 `25` 号文档里专门补了一整节，明确说明：

- `zones.yaml` 记录什么
- `gameplay_layout.yaml` 记录什么
- 两者为什么都能出现类似区域名，但语义不同

## 2.4 一次性事件示例挂错了子场景

#### 原因

早期为了先跑通新 UI 演示，我把测试事件：

- `evt_pickup_stone_axe`
- `evt_read_chopping_notice`
- `evt_stranger_reward_offer`

直接挂到了“商店街”下现成的子场景里，导致：

- “琉璃点心屋”里出现了本来不该属于它的示例事件
- “甘味茶屋”里也混入了入门流程事件

这种演示法虽然方便开发阶段快速测试，但确实会误导后续配置工作。

#### 解决方式

这次已经修改：

- `Configs/Scenarios/TestGame/UI/gameplay_layout.yaml`

现在“商店街”下新增了：

- `佐佐木木匠屋`

并且把这些示例事件统一移动到这个子场景下面：

- `evt_pickup_stone_axe`
- `evt_read_chopping_notice`
- `evt_stranger_reward_offer`

同时：

- “琉璃点心屋”现在 `event_ids: []`
- “甘味茶屋”现在 `event_ids: []`

也就是说，这几个一次性事件不会再出现在别的子场景里。

#### 进一步的保护

为了避免以后有人把同一个事件 ID 手滑挂到两个子场景里，这次还在：

- `Scripts/UI/GameplayUiConfig.cs`

里加了重复绑定检测。

现在如果同一个事件 ID 在 `gameplay_layout.yaml` 里被重复写到多个场景：

- 后面的重复绑定会被忽略
- 同时输出警告

这样可以进一步避免“某个新事件突然凭空挤到不该存在的子场景里”。

## 2.5 子场景标题排版不符合要求

#### 原因

之前子场景卡片使用的是：

- 左对齐标题
- 标题下方直接再显示一段描述文本

这与这次新要求不一致。

#### 解决方式

这次在 `Scripts/UI/GamePlayUI.Layout.cs` 里重做了子场景标题区：

- 标题改为居中显示
- 标题所在行增加一层白色半透明底色，透明度约为 `30%`
- 移除了标题下方单独显示的描述文本
- 场景描述改为通过悬浮框显示
- 场景描述悬浮时也会同步到右上角信息区

这就让子场景卡片更符合你希望的“标题更强调、描述不常驻、需要时再浮现”的结构。

## 2.6 悬浮说明没有同步到右上角信息区域

#### 原因

之前右上角的 `ConfigurableInfoPanel` 只能显示剧本默认说明内容，没有提供“临时接管显示”的能力。

所以即使某个控件本身有 tooltip：

- 右上角区域也不会跟着变化
- 鼠标离开后更谈不上恢复默认内容

#### 解决方式

这次在：

- `Scripts/UI/ConfigurableInfoPanel.cs`

里新增了：

- `SetTransientContent(...)`
- `ClearTransientContent()`

并且把它接到以下控件的鼠标悬浮事件上：

- 左侧一级区域按钮
- 左侧二级区域按钮
- 子场景标题行
- 事件按钮

现在逻辑变成：

1. 鼠标进入一个有悬浮说明的控件
2. Godot 自己弹出 tooltip
3. 右上角信息区同步切换成同一段说明
4. 鼠标离开
5. 右上角信息区恢复成剧本默认说明

这就符合你在 `32` 号文件里要求的联动效果。

## 5. 这次修正后的关键结论

你现在可以这样理解当前新 UI：

1. `zones.yaml` 不是界面层级表，它是玩法区域规则表。
2. `gameplay_layout.yaml` 才是新 UI 的一级区域 / 二级区域 / 子场景 / 事件归属表。
3. 一个事件是否显示，除了事件本身满足条件外，还必须被挂到某个子场景的 `event_ids`。
4. 现在同一个事件 ID 即使误写到多个子场景里，后面的重复绑定也会被忽略并报警。
5. 左侧区域、子场景标题、事件按钮的悬浮说明，都会同步显示到右上角信息区。
6. 按钮“点很多次才响应一次”的主因，是定时刷新时把交互控件重建掉了；这一轮已经通过延迟交互期刷新修掉。
