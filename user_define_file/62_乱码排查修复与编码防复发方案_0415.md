# 62_乱码排查修复与编码防复发方案_0415

## 1. 问题结论（先说结论）
这次“乱码”来源于两类问题：
1. 真实残留乱码：确实存在于代码字符串里，会影响游戏内显示。
2. 终端误解码假象：文件本身是 UTF-8 正常中文，但在 PowerShell 未显式指定 UTF-8 读取时，会显示成乱码，导致误判。

你看到的“编码导致上下文匹配失败”“受历史乱码影响”就是这两类问题叠加造成的。

## 3. 为什么会出现乱码

### 3.1 终端读取编码不一致（最常见）
在 Windows PowerShell 环境下，如果直接 `Get-Content`，可能按本地代码页读取 UTF-8 无 BOM 文件，导致中文显示为乱码（文件本身并未损坏）。

### 3.2 编辑写回时未显式指定 UTF-8
若写文件时未固定编码，可能出现：
- 编码变化（UTF-8 / ANSI / UTF-16 混用）
- 文本被二次错误转换

### 3.3 自动补丁上下文受乱码干扰
当工具拿到“误解码后的文本”去做上下文匹配，容易出现 patch 命中失败，或替换到错误位置。

## 4. 已落地的防复发措施

### 4.1 仓库级编码约束
已更新：`.gitattributes`
- 对 `*.cs *.yaml *.yml *.md *.txt *.json *.cfg` 明确声明 `working-tree-encoding=UTF-8`。

### 4.2 新增乱码巡检脚本
已新增：`Tools/Check-Mojibake.ps1`
能力：
1. 严格 UTF-8 校验（非法 UTF-8 直接报错）
2. 疑似乱码片段扫描（常见乱码特征）
3. 可选输出“UTF-8 无 BOM 中文文件”列表（用于判定假乱码风险）

执行命令：
```powershell
powershell -ExecutionPolicy Bypass -File .\Tools\Check-Mojibake.ps1
```

含文档一起扫描：
```powershell
powershell -ExecutionPolicy Bypass -File .\Tools\Check-Mojibake.ps1 -IncludeDocs
```

输出“UTF-8 无 BOM 中文文件”风险清单：
```powershell
powershell -ExecutionPolicy Bypass -File .\Tools\Check-Mojibake.ps1 -ReportUtf8NoBomChinese
```

## 5. 后续开发执行规范（必须遵守）

### 5.1 读取文件
- 一律使用：`Get-Content -Encoding UTF8`
- 禁止省略 `-Encoding` 读取中文内容文件。

### 5.2 写入文件
- 一律显式指定 UTF-8（建议 UTF-8 BOM 用于 PowerShell 脚本本体）。
- 禁止使用默认编码写回。

### 5.3 每次提交前
至少执行一次：
```powershell
powershell -ExecutionPolicy Bypass -File .\Tools\Check-Mojibake.ps1
```
若失败，先修编码/乱码再继续功能开发。

### 5.4 排查流程建议
1. 先确认“显示乱码”是否为终端误解码（改用 `-Encoding UTF8` 复读）。
2. 再确认是否“真实乱码写入文件”。
3. 修复后跑 `Check-Mojibake.ps1` + `dotnet build`。
