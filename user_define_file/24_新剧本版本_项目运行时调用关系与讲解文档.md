# 24_新剧本版本_项目运行时调用关系与讲解文档

## 1. 先看最短结论

当前项目运行时的主链路，可以先记成这一条：

```text
Main.tscn
  -> AppRoot.cs
    -> MainMenuUI.cs
      -> GameManager.StartScenario(...)
        -> 读取 .tres 剧本资源
        -> 读取该剧本指向的 YAML
        -> 注册物品/技能/事件/势力/区域
        -> 初始化 PlayerProfile 与各系统
          -> GamePlayUI.cs
            -> 左侧事件按钮
            -> 中间标签页
            -> 右侧状态栏
            -> 底部日志
```

如果你后面遇到问题，基本就沿着这条链往下排查。

## 2. 启动阶段是怎么走的

### 2.1 Godot 启动入口

启动时先读取：

- `project.godot`

这里决定：

- 主场景是 `Scenes/Main.tscn`
- 自动加载 `GameManager`
- 自动加载 `SignalBus`
- 视口基准是 `1920x1080`

### 2.2 自动加载单例先准备好

在真正进入主场景之前，Godot 会先初始化：

- `Scripts/Autoload/GameManager.cs`
- `Scripts/Autoload/SignalBus.cs`

其中真正关键的是 `GameManager`。

它在 `_Ready()` 里会做这些事：

1. 扫描 `Resources/Scenarios/` 下的所有 `.tres` 剧本资源
2. 建立剧本列表
3. 创建默认空玩家档案 `PlayerProfile`
4. 初始化各玩法系统节点
5. 记录一条“主菜单入口已准备完成”的运行日志

这一步结束后，程序已经知道：

- 有哪些剧本可用
- 点击“新的游戏”该进哪个默认剧本
- 点击“测试入口”该进哪个测试剧本

## 3. 主场景 `Main.tscn` 做什么

`Scenes/Main.tscn` 当前只是一个轻量外壳。

它挂载的脚本是：

- `Scripts/UI/AppRoot.cs`

`AppRoot` 负责整套界面的“大场景切换”。

你可以把它理解成：

- 主菜单和游戏内界面之间的总调度员

## 4. `AppRoot.cs` 的职责

`AppRoot` 当前主要负责四件事：

### 4.1 读取布局参数

从：

- `Resources/UI/main_ui_layout.tres`

读取 UI 基本参数，例如：

- 基准分辨率
- 左右栏宽度
- 按钮最小高度
- 对话框最小宽度
- 刷新频率

### 4.2 配置窗口缩放

当前窗口逻辑在 `ConfigureAdaptiveWindow()` 与 `UpdateAdaptiveScale()` 里。

作用是：

- 设计基准固定为 `1920x1080`
- 缩小时按比例压缩
- 放大时不超过 1 倍，避免 UI 图被放大模糊

### 4.3 默认先显示主菜单

`AppRoot._Ready()` 里最后会调用：

- `ShowMainMenu()`

它会实例化：

- `Scenes/UI/MainMenuUI.tscn`

### 4.4 响应主菜单操作

`AppRoot` 还负责处理：

- `StartDefaultStory()`
- `OpenTestGame()`
- `LoadStoryFromMenu(path)`
- `ShowMainMenu()`

也就是说，主菜单点按钮以后，真正做跳转的是它。

## 5. 主菜单 `MainMenuUI.cs` 怎么工作

主菜单脚本路径：

- `Scripts/UI/MainMenuUI.cs`

它当前主要负责：

- 显示暗色主菜单背景
- 创建主菜单按钮
- 打开正式存档读取弹窗
- 提供右上角绿色“进入游戏测试”按钮

它并不自己决定“读哪套剧本”，而是把选择交回 `AppRoot`。

也就是：

- 点“新的游戏” -> `AppRoot.StartDefaultStory()`
- 点“进入游戏测试” -> `AppRoot.OpenTestGame()`
- 点“读取存档” -> `AppRoot.LoadStoryFromMenu(path)`

## 6. `GameManager.StartScenario()` 是真正的剧本切换核心

不管是进入主线，还是进入测试剧本，最后都会走到：

- `GameManager.StartScenario(string scenarioId, string savePath, bool resetProfile)`

这个方法可以说是整个“新剧本版本”的中枢。

它主要做这些事：

1. 根据 `scenarioId` 找到对应的 `GameScenarioDefinition`
2. 把它记为当前活动剧本 `ActiveScenario`
3. 视情况重置玩家档案
4. 设置当前存档路径
5. 调用 `LoadStaticData(scenario)` 读取这套剧本的静态数据
6. 调用 `InitializeProfileFromDefinitions()` 给运行时档案补齐默认状态
7. 调用 `ConfigureSystems()` 把注册表绑定给各系统
8. 写入一条“已载入剧本”的日志

## 7. 剧本静态数据是怎么读进来的

`GameManager.LoadStaticData(scenario)` 会根据 `.tres` 里写的路径去读 YAML。

例如：

- `ItemsConfigPath`
- `SkillsConfigPath`
- `OneshotEventsConfigPath`
- `ClickEventsConfigPath`
- `IdleEventsConfigPath`
- `FactionsConfigPath`
- `ZonesConfigPath`
- `LocalizationPath`

读取动作由：

- `Scripts/Core/Helpers/YamlConfigLoader.cs`

完成。

然后分别装入：

- `ItemRegistry`
- `SkillRegistry`
- `EventRegistry`
- `FactionRegistry`
- `ZoneRegistry`
- `LocalizationManager`

所以：

- `.tres` 决定“读哪套配置”
- `YamlConfigLoader` 负责“把配置读出来”
- `Registry` 负责“把数据注册好，供系统查询”

## 8. 为什么主线和测试剧本现在终于分开了

以前的问题，不是在 UI，而是在这里：

- 两个剧本资源虽然存在
- 但它们都还在指向同一套旧配置

所以运行时走到 `LoadStaticData()` 的时候，不管你选主线还是测试，装进来的都是石斧测试内容。

现在修正后：

- 主线剧本读 `Configs/Scenarios/MainStory/...`
- 测试剧本读 `Configs/Scenarios/TestGame/...`

因此这一步就已经真正分流了。

## 9. 游戏内主界面 `GamePlayUI.cs` 做什么

剧本加载成功后，`AppRoot` 会切换到：

- `Scenes/UI/MainUI.tscn`

这个场景现在只是一个壳。

真正逻辑在：

- `Scripts/UI/GamePlayUI.cs`

它是当前游戏内界面的总装配脚本。

### 9.1 左侧事件区

它会从 `EventRegistry` 里按 `ButtonListGroup` 分组取事件：

- `MainOneshot`
- `MainClick`
- `MainIdle`

然后分别显示到三列：

- 一次性事件
- 单次点击事件
- 挂机事件

### 9.2 中间标签页

当前中间区会创建这些标签页：

- 背包
- 技能
- 图鉴
- 势力
- 区域
- 系统（仅正式剧本）

其中：

- 背包页由 `InventoryPanel.cs` 刷新
- 技能页由 `SkillPanel.cs` 刷新
- 图鉴页由 `DictionaryPanel.cs` 刷新
- 势力页由 `FactionPanel.cs` 刷新
- 区域页由 `ZoneSelectPanel.cs` 刷新
- 系统页由 `SystemPanel.cs` 提供正式存读档与返回主菜单入口

### 9.3 右侧状态与提示

右侧主要显示：

- 当前剧本名称
- 引导提示
- 角色状态栏

状态栏由 `CharacterStatusPanel.cs` 刷新，里面会读取：

- 金币
- 当前工具
- 技能等级与经验
- 当前挂机状态

### 9.4 底部日志区

底部日志由 `LogPanel.cs` 显示。

它读取的是：

- `GameManager.RuntimeLogs`

也就是说，很多系统层写进来的日志，最后都会显示到这里。

## 10. 点击事件和挂机事件是怎么流转的

### 10.1 点击事件

左侧按钮按下后，`GamePlayUI` 会查当前事件定义。

如果是普通点击事件，最终会走到：

- `ClickEventSystem.TryTriggerEvent(eventId)`

它负责：

1. 判断事件是否存在
2. 判断一次性事件是否已经消耗
3. 判断前置条件是否满足
4. 判断消耗是否足够
5. 支付消耗
6. 发放奖励与效果
7. 如有需要，把事件记入 `CompletedEventIds`

### 10.2 带弹窗的一次性事件

如果这个事件配置了 `dialog`，那么流程会先停在 UI 层：

- `GamePlayUI.ShowDialogForEvent(...)`

它会把弹窗按钮组装出来，然后根据选择：

- 执行确认按钮事件
- 或执行目标分支事件
- 或点“我再想想”直接关闭，不消耗一次性事件

### 10.3 挂机事件

挂机按钮按下后，`GamePlayUI` 会调用：

- `IdleSystem.StartIdleEvent(eventId)`

之后真正的进度推进发生在：

- `IdleSystem._Process(double delta)`

它会累计时间，到达阈值后：

- 发放产出物
- 发放技能经验
- 写入日志
- 继续下一轮挂机循环

## 11. 技能经验和升级是怎么走的

### 11.1 经验来源

当前技能经验主要来自两类：

- 事件效果 `GrantSkillExp`
- 挂机事件结算时自动发放经验

### 11.2 升级处理

技能页点击“手动升级”后，会调用：

- `SkillSystem.TryLevelUp(skillId)`

它负责：

1. 找到当前技能定义
2. 看当前等级是否还能升
3. 看当前存储经验是否满足升级门槛
4. 扣除升级所需经验
5. 提升等级
6. 刷新 `CanLevelUp`

## 12. 存档系统怎么流转

### 12.1 测试剧本快捷存读档

测试剧本进入游戏内界面后，会在底部保留快捷按钮：

- 保存存档
- 读取存档

这套默认走的是：

- `save_data.json`

### 12.2 正式剧本 10 槽位存读档

正式剧本则通过“系统”标签页进入：

- 保存存档
- 读取存档
- 返回主菜单

正式剧本的 10 槽位路径由：

- `RuntimePathHelper.GetStorySaveSlotPath(slotIndex)`

生成，也就是：

- `save_01.json`
- `save_02.json`
- ...
- `save_10.json`

### 12.3 存读档底层执行者

真正写 JSON 的是：

- `Scripts/Core/SaveLoad/SaveManager.cs`

它负责：

- 把 `PlayerProfile` 转成可序列化结构
- 写入 JSON
- 读取 JSON
- 恢复为运行时档案
- 读取存档元信息列表

## 13. “返回主菜单”现在是怎么工作的

正式剧本“系统”标签页里的“返回主菜单”按钮，最终会走到：

- `GamePlayUI.RequestReturnToMainMenu()`

它会先弹确认框。

玩家确认后，再去找到：

- `AppRoot`

然后调用：

- `AppRoot.ShowMainMenu()`

也就是说，返回主菜单是 UI 层主动请求总控切回主菜单，而不是重新启动整个游戏。

## 14. UI 图片主题在运行时怎么被用到

这部分流程是：

```text
Configs/UI/ui_image_theme.yaml
  -> UiImageThemeManager.LoadTheme()
    -> MainMenuUI / EventButtonItem / SystemPanel / GamePlayUI
```

当前已经接入的主要对象：

- 主菜单背景
- 主菜单按钮
- 一次性事件按钮
- 点击事件按钮
- 挂机事件按钮
- 系统页按钮
- 测试模式特殊按钮

也就是说，你后面主要换图时，不需要先改 UI 布局逻辑，先改 `ui_image_theme.yaml` 即可。

## 15. 运行时调试时最常见的排查顺序

如果你以后遇到“按钮不对 / 剧本不对 / 内容不对”，我建议按这个顺序排查：

1. 看 `Resources/Scenarios/*.tres` 是否指向了正确 YAML
2. 看对应 `Configs/Scenarios/...` 里的 YAML 是否真的写了这些内容
3. 看 `GameManager.LoadStaticData()` 是否已经把这些内容读进注册表
4. 看 `GamePlayUI` 是否把相应事件分组显示出来
5. 再看具体系统逻辑是否执行成功

## 16. 用一句话概括当前运行时架构

你可以把现在的架构理解成：

- `AppRoot` 管场景切换
- `GameManager` 管剧本加载和系统初始化
- `Registry` 管静态定义查询
- `Systems` 管实际玩法执行
- `GamePlayUI` 管游戏内界面展示与交互
- `SaveManager` 管持久化

这就是当前“新剧本版本”的完整运行时骨架。
