主要 findings 如下，按严重度排序。

- 高：运行时写入路径设计放在 `res://`，导出后会失效。文档把日志输出目录定义在 `res://Logs`，并且注册表/校验流程会在启动时打印到文件；这在 Godot 编辑器里能跑，但导出包里 `res://` 是只读的，最终会变成“开发环境正常，发布版失败”。

- 高：挂机状态出现双重真相源，而且没有定义离线结算契约。`PlayerProfile` 持久化了 `ActiveIdleEventId` 和 `IdleAccumulatedTime`，`IdleSystem` 自己又维护了一套 `CurrentIdleEventId` / `AccumulatedTime` / `CurrentActionInterval`。这会直接带来读档恢复、切场景、暂停恢复时的同步问题；同时文档虽然有 `LastSaveTime` / `SaveTimestamp`，但没有定义离线收益如何重放，这对挂机游戏是核心缺口。

- 高：版本与迁移模型不自洽。`SaveVersion` 同时存在于 `PlayerProfile` 和 `SaveData`，保存时还要额外传一个 `version` 参数，这意味着版本号有 3 个入口，后续很容易不一致。更严重的是 `MigrationRunner` 用 `SortedDictionary<string, Action<SaveData>>` 管迁移，按字符串排序语义版本是有坑的，且它只显式表达了 `fromVersion`，没有表达文档 YAML 里的 `from -> to` 链。

- 高：配置接口过度字符串化，已经出现模型不一致。`ItemTag` 是强类型枚举，但 `SkillDefinition.RequiredToolTag` 却是字符串；样例里直接写了 `"axe"`，而枚举里并没有这个值。类似问题还出现在 `ConditionType`、`ToolBonuses`、`ButtonListGroup`、`ExtensionData` 等位置。这样做会让“配置驱动 + 启动时校验”这个目标打折，因为大量错误只能跑到运行时才暴露。

- 中：经济系统缺少明确的货币运行态与商店库存持久化。你定义了 `CurrencyType.Gold`、`GoldCost`、`SellSystem` 的卖出金币收益，也给商店商品加了 `stock`，但 `PlayerProfile` 里没有钱包状态，也没有任何“已售罄/剩余库存”的玩家态或世界态。现在这个设计下，金币到底是独立数值还是某个 item，没有落地；`stock: 1` 这种限量商品也没有地方持久化。

- 中：事件系统的“效果面”过窄，后续很容易退化成按事件 ID 写硬编码。`EventDefinition` 目前只能表达物品消耗、物品奖励和技能经验，但文档同时又声明了 faction、zone、quest、peace、battle 等系统，还允许 faction 绑定 event。问题是这些系统的状态变化并没有进入统一 effect 模型，所以一旦出现“点击事件加声望”“完成事件解锁区域”这类需求，只能在系统里针对事件 ID 特判。

- 中：排序职责混在“静态定义层”和“玩家显示层”之间，边界不清。总结里写的是 `definition_order + player_display_order` 双层排序，但 `ItemRegistry` 又提供了运行时改定义顺位的 `MoveNodeOrder/MoveNodeBefore`，示例 YAML 甚至直接建议调用 registry 去改馒头的排序。这样会把本应稳定的配置定义变成运行时可变状态，后续存档、迁移、多人配置协作都会变乱。

- 中：信号体系有重复定义，后面会出现“到底订阅谁”的问题。文档一方面说所有系统通过 `SignalBus` 发信号，另一方面每个系统自己又声明了一套 delegate。这个模式很容易演化成双通道通知，UI 有的地方订系统实例，有的地方订总线，最后难排查重复刷新和漏刷新。

整体评价：这份设计的方向是对的，尤其是“配置驱动”“模板/运行态分离”“注册表集中校验”这三点都适合 Godot + C#。但目前最需要先收紧的是 4 个边界：状态唯一真相源、版本迁移模型、强类型配置契约、经济/事件的运行态表达。它们不先定清楚，后面越往 UI 和内容层推进，返工会越大。
