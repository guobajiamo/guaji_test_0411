# 21_新剧本配置相关说明文档

## 先说结论

现在项目已经不是“把所有配置都硬写死在一套代码里”了，而是改成了：

- `GameManager` 负责扫描剧本
- 每个剧本用一个 `.tres` 资源描述
- 每个剧本再指向自己要使用的物品、事件、技能、区域、文本等配置文件

你以后如果想加“新主线”“外传”“活动剧本”“测试剧本 2 号”，主要就是新增一个剧本资源，然后决定它要引用哪一套配置文件。

## 现在剧本配置放在哪里

剧本资源目录：

- `Resources/Scenarios/`

当前已有两个剧本：

- `Resources/Scenarios/main_story_scenario.tres`
- `Resources/Scenarios/test_game_scenario.tres`

它们都会被 `GameManager` 启动时自动扫描到。

## 默认新游戏剧本是怎么指定的

默认“新的游戏”按钮进入哪个剧本，主要看剧本资源里的这个字段：

- `IsDefaultStoryScenario = true`

当前默认新游戏使用的是：

- `Resources/Scenarios/main_story_scenario.tres`

因为它被设置成了：

- `IsDefaultStoryScenario = true`

而测试剧本没有这个标记，所以不会被当作“新的游戏”的默认入口。

## 剧本资源里每个字段大概是什么意思

下面用当前的 `main_story_scenario.tres` 说明：

### 1. 身份字段

- `ScenarioId`
  这是剧本的内部唯一 ID，不能和别的剧本重复。

- `DisplayName`
  这是显示给玩家看的剧本名称。

- `Description`
  这是给开发阶段或以后菜单说明用的描述文字。

### 2. 菜单与入口控制字段

- `ShowInNewGameMenu`
  这个剧本是否应该出现在“新游戏候选列表”里。

- `IsDefaultStoryScenario`
  这个剧本是否是当前“新的游戏”的默认正式主线。

- `IsTestScenario`
  这个剧本是不是测试入口专用剧本。

- `EnableSystemTab`
  进入这个剧本后，游戏内中间标签页是否显示 `系统` 标签。

### 3. 数据配置路径字段

- `CategoriesConfigPath`
  物品分类配置。

- `ItemsConfigPath`
  物品定义配置。

- `SkillsConfigPath`
  技能配置。

- `OneshotEventsConfigPath`
  一次性事件配置。

- `ClickEventsConfigPath`
  单次点击事件配置。

- `IdleEventsConfigPath`
  挂机事件配置。

- `FactionsConfigPath`
  势力与 NPC 配置。

- `ZonesConfigPath`
  区域配置。

- `LocalizationPath`
  文本本地化配置。

## 为什么要这样拆

这样拆的好处是，你以后做不同剧本时，不需要改一大堆代码逻辑，只需要改“这个剧本引用哪套配置”。

例如：

- 剧本 A 用一套物品表
- 剧本 B 用另一套技能表
- 剧本 C 用不同的区域和事件树

底层系统仍然是同一套，但剧本实际内容不同。

这正好符合你之前提出的需求：

- 不同剧本可能有不同道具
- 同名道具可能有不同效果
- 不同剧本可能有不同区域
- 不同剧本可能有不同技能树和经验体系
- 不同剧本可能有不同货币体系

## 以后如果你要新增一个正式剧本，该怎么做

下面给你一个最稳妥的做法。

### 第一步：复制一个已有剧本资源

最简单的方法是复制：

- `Resources/Scenarios/main_story_scenario.tres`

然后改名，例如：

- `Resources/Scenarios/forest_story_scenario.tres`

### 第二步：修改剧本身份字段

至少要改下面这些：

- `ScenarioId`
  例如改成 `forest_story`

- `DisplayName`
  例如改成 `森林支线剧本`

- `Description`
  写清楚这个剧本是做什么的

### 第三步：决定它是不是默认主线

如果你希望它成为新的默认新游戏入口：

- 把它设为 `IsDefaultStoryScenario = true`
- 并把原来的默认主线改成 `false`

如果你只是想新增一个备用剧本，不抢默认入口：

- 保持 `IsDefaultStoryScenario = false`

### 第四步：决定它要不要出现在未来的新游戏列表里

如果以后你想让玩家能在“新游戏”里看到它：

- `ShowInNewGameMenu = true`

如果它只是开发中隐藏剧本：

- `ShowInNewGameMenu = false`

### 第五步：决定它引用哪套配置

你可以直接复用旧配置，也可以给它做新配置文件。

例如：

- 物品改成：`res://Configs/Items/items_forest.yaml`
- 技能改成：`res://Configs/Skills/skills_forest.yaml`
- 一次性事件改成：`res://Configs/Events/oneshot_events_forest.yaml`
- 区域改成：`res://Configs/Zones/zones_forest.yaml`

这样这个新剧本就会读自己的那一套内容。

## 如果我只想改“默认主线剧本包含哪些内容”，该改哪里

你可以直接改当前默认主线剧本引用的这些配置文件：

- `Configs/Items/categories.yaml`
- `Configs/Items/items.yaml`
- `Configs/Skills/skills.yaml`
- `Configs/Events/oneshot_events.yaml`
- `Configs/Events/click_events.yaml`
- `Configs/Events/idle_events.yaml`
- `Configs/Factions/factions.yaml`
- `Configs/Zones/zones.yaml`
- `Configs/Localization/locale_zh.yaml`

但要注意：

- 如果你直接改这些文件，默认主线剧本和测试剧本现在都会一起受影响
- 因为目前这两个剧本暂时还是共用同一套配置包

## 如果我想让“默认主线”和“测试剧本”彻底分开，应该怎么做

做法也很简单：

1. 复制现有配置文件，做一份测试专用配置
2. 修改 `test_game_scenario.tres`，让它改为引用测试专用配置
3. 保持 `main_story_scenario.tres` 继续引用正式主线配置

例如以后你可以做成这样：

- 主线剧本使用：
  - `items_main_story.yaml`
  - `skills_main_story.yaml`
  - `zones_main_story.yaml`

- 测试剧本使用：
  - `items_test.yaml`
  - `skills_test.yaml`
  - `zones_test.yaml`

这样测试剧本里再乱改，也不会影响正式主线。

## 当前“新游戏”为什么只会进一个固定剧本

因为这次先按你的要求，做的是：

- 先保留默认单入口
- 但底层已经预留多剧本能力

也就是说：

- 现在按钮行为还是“点了就进默认主线”
- 但内部已经不是写死配置了
- 以后只要扩展主菜单 UI，就可以把多个 `ShowInNewGameMenu = true` 的剧本列出来供玩家选择

## 你后续最常改的几个位置

如果你以后继续自己调内容，最常接触的大概率会是下面几个地方：

看这里：

- `Resources/Scenarios/main_story_scenario.tres`
- 以及其他剧本 `.tres` 里的 `IsDefaultStoryScenario`

### 想改主线剧本里有哪些事件

### 想改主线剧本里有哪些物品

### 想改主线剧本里有哪些技能

## 给你的一个实用建议

如果你准备正式开始做“多个剧本”，我建议你从下一次开始就把“主线配置”和“测试配置”分开，不要继续共用同一套 YAML。

原因很简单：

- 共用配置时，测试改动很容易误伤正式内容
- 分开以后，测试剧本可以大胆做临时事件和临时数值
- 正式主线会更稳定，也更方便排查问题

## 当前实现和你后续扩展之间的关系

你可以把这次实现理解成一句话：

“先把多剧本地基打好，现在默认只摆一扇门，之后再继续往上加房间。”

也就是说，这次最重要的不是菜单按钮本身，而是已经把下面这些基础打通了：

- 主菜单 -> 进入不同剧本
- 不同剧本 -> 指向不同配置包
- 不同剧本 -> 使用同一套系统逻辑
- 正式故事 -> 使用 10 槽位存档
- 测试入口 -> 继续使用旧快捷存档
