# 45_任务系统实装需求（补充更新：货币实体化）

## 1. 本次补充目的

在保留原有“金币作为交易货币”功能基础上，新增一条强约束：

- 货币必须同时是背包中的实体道具。

也就是任意货币都具备“双重身份”：

1. 可用于买卖与条件判断（货币语义）
2. 可作为物品堆叠在背包中（道具语义）

## 2. 实装结果（代码侧）

### 2.1 新增统一数值结算中间层

- `Scripts/Core/Helpers/ValueSettlementService.cs`

职责：

- 统一处理道具、货币、经验的结算
- 统一处理买卖价格、挂机速度/产出、经验倍率
- 统一执行“货币账本 <-> 背包实体”同步

### 2.2 货币实体映射

默认金币映射为：

- `gold` -> `currency_gold`

扩展货币映射规则：

- `currency_{currencyId}`

例如未来若有 `jade_token`，其实体道具 ID 可为：

- `currency_jade_token`

### 2.3 已接入统一结算的系统

- `IdleSystem`
- `ShopSystem`
- `SellSystem`
- `ClickEventSystem`
- `QuestSystem`

这些系统不再直接改库存/金币/经验，而是调用中间层接口执行结算。

## 3. 对任务系统的影响

### 3.1 任务奖励中的金币发放/扣除

任务奖励效果 `GrantGold` / `RemoveGold` 现在通过中间层执行：

- 仍会改变货币账本
- 同时会同步到背包中的 `currency_gold` 堆叠

### 3.2 任务解锁条件的兼容性

`HasGold` 条件继续读取经济账本，逻辑与旧版一致，不影响既有任务 YAML。

由于账本与背包实体实时同步，玩家在 UI 层也能看到货币作为道具存在。

## 4. 对交易与挂机的联动影响

### 4.1 交易

- 商店买入价格支持统一倍率入口（折扣/加价）
- 卖出收益支持统一倍率入口
- 支付与到账会同步背包货币实体

### 4.2 挂机

- 挂机收获总量、速度、经验全部通过中间层计算
- 未来限时 Buff（产量加成、速度提速）可直接接入倍率键

## 6. 使用约束（后续开发）

后续新增数值逻辑时，必须优先走中间层，不要在业务系统里直接：

- `Inventory.AddItem/TryRemoveItem`
- `Economy.AddGold/TrySpendGold`
- `SkillSystem.AddExp`

统一通过 `ValueSettlementService` 可避免“即时结算分散在多处”导致的规则漂移问题。
