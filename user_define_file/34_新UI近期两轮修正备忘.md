# 34_新UI近期两轮修正备忘

## 1. 这份备忘是干什么的

这份文档是给后续继续改新 UI 时快速回忆上下文用的。

主要记录的是最近两轮新 UI 修正里，已经明确做过、并且以后很容易再次碰到的几个关键点：

- 新 UI 的区域层级和事件归属到底怎么理解
- 为什么主界面刷新机制从定时轮询改成了事件驱动
- 悬浮说明和右上角信息区联动时要注意什么
- 滚动条逻辑目前已经覆盖到哪些面板
- 事件按钮为什么必须统一结构，不能因为事件类型不同而参差不齐

## 2. 当前新 UI 的职责划分

### 2.1 `zones.yaml` 不是界面树结构表

`Configs/Scenarios/*/Zones/zones.yaml` 负责的是：

- 区域玩法定义
- 通关次数
- 解锁链
- 解锁条件

也就是说，它属于：

- 探索/副本/战斗/解锁规则层

### 2.2 `gameplay_layout.yaml` 才是新 UI 的内容布局表

`Configs/Scenarios/*/UI/gameplay_layout.yaml` 负责的是：

- 一级区域
- 二级区域
- 子场景
- 某个事件按钮显示在哪个子场景

也就是说，它属于：

- 左侧区域树
- 当前区域页的子场景卡片
- 事件归属位置

以后如果再看到“人间之里 / 商店街 / 琉璃点心屋 / 佐佐木木匠屋”这种层级名，不要第一反应就往 `zones.yaml` 里塞，要先判断它到底是在表达：

- 玩法区域规则
- 还是 UI 层级布局

## 3. 事件显示位置的当前规则

事件是否能在新 UI 里被玩家看到，必须同时满足两件事：

1. 事件本身在 `Events/*.yaml` 里被定义出来
2. 该事件 ID 被写进某个 `scene.event_ids`

如果只定义了事件本身，但没挂到 `gameplay_layout.yaml` 的某个子场景里，它不会自动显示。

### 3.1 现在已经加了“重复挂载保护”

当前 `Scripts/UI/GameplayUiConfig.cs` 在读取 `gameplay_layout.yaml` 时，会检查：

- 同一个事件 ID 是否被写到了多个子场景

如果重复了：

- 后面的重复挂载会被忽略
- 同时输出警告

这样是为了避免：

- 一个事件莫名其妙同时出现在多个不该出现的子场景里

## 4. 测试剧本里入门示例事件现在挂在哪里

之前这几个一次性事件曾经被临时挂在不合适的子场景里，后来已经统一整理过。

当前测试剧本里：

- `evt_pickup_stone_axe`
- `evt_read_chopping_notice`
- `evt_stranger_reward_offer`

都挂在：

- `人间之里 -> 商店街 -> 佐佐木木匠屋`

而不是：

- `琉璃点心屋`
- `甘味茶屋`

如果后续还要继续加“入门引导 / 教程 / 木工 / 伐木前置”的示例事件，优先继续往“佐佐木木匠屋”这个子场景里放，不要再塞回别的展示性子场景。

## 5. 为什么主界面刷新机制改成了事件驱动

### 5.1 旧问题

之前 `GamePlayUI` 用 `_Process` 做定时轮询刷新。

问题在于：

- UI 每隔固定时间就会整页刷新
- 刷新时会重建一些交互控件
- 如果玩家刚好在点击按钮，就可能出现按钮被刷新中途重建掉

直接后果就是：

- 按钮经常需要点很多次才响应
- 悬浮说明刚显示出来又被默认内容抢回去

### 5.2 现在的做法

现在主界面不再依赖定时轮询来刷新。

当前改成：

- 绝大多数刷新由玩家操作直接触发
- 挂机状态变化这类后台变化，通过 `SignalBus.ActiveIdleEventChanged` 触发刷新

相关文件：

- `Scripts/UI/GamePlayUI.cs`
- `Scripts/Systems/IdleSystem.cs`
- `Scripts/Autoload/SignalBus.cs`

### 5.3 为什么悬浮期间还要做“延后刷新”

即使改成事件驱动，也仍然会遇到一种情况：

- 鼠标正停在一个有悬浮说明的控件上
- 后台恰好发生了挂机结算或状态变化
- 如果此时立刻整页刷新，右上角信息区还是会被抢回默认内容

所以现在做法是：

- 如果右上角信息区当前正处于“悬浮内容接管”状态
- 后台刷新请求先记为“待刷新”
- 等鼠标离开悬浮目标后，再补做一次完整刷新

这样既能保留悬浮期间的阅读稳定性，又不会漏掉后台状态变化。

## 6. 悬浮说明联动的当前规则

目前这些控件都已经接入了统一的悬浮说明联动：

- 左侧一级区域按钮
- 左侧二级区域按钮
- 当前区域页的子场景标题行
- 事件按钮

联动方式是：

1. Godot 自己弹出 tooltip
2. 右上角 `ConfigurableInfoPanel` 同步显示相同说明
3. 鼠标离开后，右上角恢复成剧本默认信息

相关文件：

- `Scripts/UI/ConfigurableInfoPanel.cs`
- `Scripts/UI/GamePlayUI.Layout.cs`

如果以后要新增“悬浮说明 + 右上角同步”的新控件，优先复用 `BindHoverInfo(...)` 这套接法。

## 8. 为什么事件按钮必须统一结构

### 8.1 出过什么问题

之前“点击事件”和“挂机事件”在同一排里会看起来不齐。

表面上像是按钮高度不一致，但真正原因通常是：

- 挂机事件前面多了一条进度条
- 普通点击事件前面没有这个占位
- 结果同一排里一个按钮被往下顶，另一个没有

用户主观上就会觉得：

- 两个按钮不齐
- 一排布局不统一

### 8.2 现在怎么处理

现在每个事件格子统一采用同一套结构：

1. 上方先保留一个固定高度的“状态槽位”
2. 挂机事件把进度条放进这个槽位
3. 普通点击事件这个槽位留空，但仍然保留同样高度
4. 下面再放统一高度的按钮

这样就能保证：

- 一排两个按钮的对齐方式一致
- 不会因为某个事件类型特殊，就把整排布局搞得参差不齐

相关文件：

- `Scripts/UI/GamePlayUI.Layout.cs`

### 8.3 后来为什么又出现“进度条不见了”

后面做“所有事件按钮等高”时，曾经临时把进度条塞进一个不会自动排版子节点的普通 `Control` 里。

结果就是：

- 高度对齐了
- 但上方进度条没有被正确铺出来

同时，因为主界面已经改成“事件驱动刷新”，如果没有额外补一层“轻量进度值更新”，右侧人物状态栏里的目标进度条也不会持续变化。

### 8.4 现在的最终做法

现在已经把事件按钮正式收口成统一组件：

- `Scripts/UI/EventButtonItem.cs`

这个组件固定包含两层：

1. 上方进度条槽位
2. 下方按钮本体

所有事件类型都复用它：

- 一次性事件
- 单次点击事件
- 挂机事件

区别只在于：

- 一次性事件 / 点击事件：进度条槽位保留，但进度条隐藏
- 挂机事件：执行中显示进度条，不执行时隐藏

这样可以同时保证：

- 布局统一
- 等高一致
- 不再因为临时拼装进度条而错位或消失

## 8.5 进度更新的当前原则

主界面现在不再靠“整页定时轮询刷新”来更新进度。

而是拆成两种：

1. 整页内容刷新：
   由玩家操作或挂机状态切换/结算这类事件触发
2. 进度条数值刷新：
   只做轻量更新，不重建 UI

也就是说：

- 允许进度条自己持续动
- 但不允许为了让它动，就把整页 UI 恢复成高频重建

## 8.6 挂机与普通事件的并行规则

当前逻辑已经明确保证：

- 一次性事件不会打断挂机
- 单次点击事件不会打断挂机
- 只有两种情况会中断或切换当前挂机

分别是：

1. 玩家再次点击当前正在运行的那个挂机事件，表示主动取消挂机
2. 玩家点击另一个新的挂机事件，表示切换挂机目标

这个规则目前已经加进自动化冒烟测试里，避免以后改 UI 或改事件系统时被意外破坏。

## 11. 最后一句话总结当前原则

当前这套新 UI 后续继续扩时，尽量遵守这三条：

1. 内容归属优先靠 `gameplay_layout.yaml` 管，不要再让事件“凭空出现”。
2. 主界面刷新优先走“事件触发”，不要轻易再加回定时轮询。
3. 同类交互控件必须共用统一骨架，不能因为事件类型不同就把按钮排版搞散。
## 12. 新增区域显隐前置条件系统

这一次又补了一层很重要的新规则：

- 一级区域可以配显示前置条件
- 二级区域可以配显示前置条件
- 子场景也可以配显示前置条件

统一都写在：

- `Configs/Scenarios/*/UI/gameplay_layout.yaml`

统一字段是：

- `visibility_condition_count`
- `visibility_conditions`

### 12.1 当前支持的条件来源

这套显隐条件复用了通用条件判定器，所以它和事件前置条件是同一套条件系统。

当前已经支持：

- 背包是否有某个道具 `HasItem`
- 是否完成过某个一次性事件 `EventCompleted`
- 是否通关过某个副本/区域 `ZoneCleared`
- 以及 `HasGold / SkillLevel / FactionReputation / QuestCompleted`

### 12.2 条件数量和条件条目数会做一致性检查

现在加载 `gameplay_layout.yaml` 时，会检查：

- `visibility_condition_count`
- `visibility_conditions` 实际条目数

是否一致。

如果不一致，会打印错误提示，方便排查 YAML。

### 12.3 父级隐藏会连带隐藏下级

这次的显示逻辑不是各层独立判断，而是：

- 一级区域隐藏时，下面的二级区域不会显示
- 二级区域隐藏时，下面的子场景不会显示

即使下级自己的条件满足，只要父级还没显示，它也不会越级跳出来。

### 12.4 显隐条件依赖的历史数据已经会进存档

这次没有额外发明一套新的“区域解锁存档字段”，而是直接复用了已有存档历史：

- `PlayerProfile.Inventory`
- `PlayerProfile.CompletedEventIds`
- `PlayerProfile.CompletedQuestIds`
- `PlayerProfile.ZoneStates[*].ClearCount`

所以玩家读档后不会丢失已经满足过的解锁历史。

### 12.5 当前测试剧本里专门保留的两个显隐示例

为了后续继续调试，这次特意保留了两个例子：

- `旧地狱`
  现在一级区域默认隐藏，需要占位事件 `evt_unlock_old_hell_placeholder`
- `迷途竹林 -> 永远亭`
  `迷途竹林` 默认可见，但 `永远亭` 需要完成 `evt_talk_to_rabbit_chief`

其中 `evt_talk_to_rabbit_chief` 旁边已经写了注释，提醒这条事件还没真正实例化。

### 12.6 这次顺手修掉的一个隐藏配置坑

一开始 `永远亭` 那条 TODO 注释和真正的条件条目挤在了同一行。

这种写法很危险，因为：

- YAML 解析时可能把后面的条件当成注释内容吃掉
- 最终就会变成 `visibility_condition_count` 写了 1
- 但真正读出来的条件列表可能是 0

现在已经改成：

- 注释单独一行
- 条件条目单独下一行

以后在 YAML 里写“TODO 注释 + 列表项”时，尽量也保持这种写法，避免出现同类问题。
