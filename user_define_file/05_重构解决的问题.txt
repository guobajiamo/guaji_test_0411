《本次 review 后已解决的问题汇总》

说明：
这份文档对应我本次根据 `02_v0.4版本最新提示词汇总版.txt`、`03_v0.4版本claude输出架构（仅声明与备注）.txt`、`04_review及相关提示.txt` 对当前项目骨架做的代码审查和修改。
目标不是把整个游戏一次性做完，而是先把“底层骨架里最容易后面返工的大问题”提前收紧。

==============================
一、本次 review 的核心结论
==============================

这次主要集中解决了 8 类问题：

1. 运行时写入路径不能继续依赖 `res://`
2. 挂机状态不能有两份“真相源”
3. 存档版本号和迁移链需要统一入口
4. 配置接口不能过度字符串化
5. 金币与商店库存需要独立运行态
6. 事件系统需要更宽的 effect 表达层
7. 排序职责要从“定义层”和“玩家显示层”拆开
8. 补上本轮排查时发现的几个具体运行 bug / 编译风险


========================================
二、已解决的问题、解决方式、修改文件
========================================

1. 已解决：运行时写入路径放在 `res://` 的问题

问题说明：
Godot 导出后，`res://` 基本可以理解成只读资源区。之前如果把存档、日志、导出的树结构文件都往 `res://` 写，编辑器里可能没问题，但导出版本会失败。

解决方式：
- 新增 `RuntimePathHelper`，统一管理运行时可写路径
- 存档路径改成 `user://save_data.json`
- 日志目录改成 `user://logs`
- 物品树导出改成 `user://logs/item_tree_dump.txt`
- `TreePrinter.WriteToFile()` 在写文件前会先确保父目录存在

主要修改文件：
- `Scripts/Core/Helpers/RuntimePathHelper.cs`
- `Scripts/Core/Helpers/TreePrinter.cs`
- `Scripts/Core/Registry/ItemRegistry.cs`
- `Scripts/Core/SaveLoad/SaveManager.cs`

补充说明：
项目里现在仍然保留了 `Logs/item_tree_dump.txt` 这个旧文件，它更像开发期遗留样本，不再应该作为正式运行时输出目标。


2. 已解决：挂机状态双重真相源、缺少离线结算契约的问题

问题说明：
原本设计里，`PlayerProfile` 和 `IdleSystem` 各自维护一套挂机状态，后面一旦出现切场景、读档、暂停恢复，就很容易不同步。

解决方式：
- 新增 `PlayerIdleState` 作为唯一挂机运行态
- `PlayerProfile` 只保留一份 `IdleState`
- `IdleSystem` 改成只读写 `PlayerProfile.IdleState`
- 明确补上 `ApplyOfflineProgress(long nowUnixSeconds)` 方法，先把“离线补算的输入输出契约”定下来

主要修改文件：
- `Scripts/Core/Runtime/PlayerIdleState.cs`
- `Scripts/Core/Runtime/PlayerProfile.cs`
- `Scripts/Systems/IdleSystem.cs`
- `Scripts/Autoload/GameManager.cs`


3. 已解决：存档版本号入口重复、迁移模型不自洽的问题

问题说明：
之前 `SaveVersion` 分散在多个位置，很容易出现“这里改了，那里没改”的情况；迁移器如果只按字符串排序，也会埋下版本比较错误。

解决方式：
- 新增 `SemanticVersion`，把版本比较改成明确的 `major.minor.patch`
- 新增 `SaveMetadata`，把版本号、保存时间这类技术字段集中放到 `SaveData.Metadata`
- `PlayerProfile` 不再持有自己的 `SaveVersion`
- `MigrationRunner` 改成显式 `FromVersion -> ToVersion`
- `SaveManager` 统一在保存时写入 `Metadata.SaveVersion`

主要修改文件：
- `Scripts/Core/SaveLoad/SemanticVersion.cs`
- `Scripts/Core/SaveLoad/SaveMetadata.cs`
- `Scripts/Core/SaveLoad/SaveData.cs`
- `Scripts/Core/SaveLoad/MigrationRunner.cs`
- `Scripts/Core/SaveLoad/SaveManager.cs`
- `Scripts/Autoload/GameManager.cs`


4. 已解决：配置接口过度字符串化的问题

问题说明：
如果 `required_tool_tag`、`condition_type`、`button_list_group` 这些关键配置全是字符串，拼错一个字母就要拖到运行时才发现，而且和“启动时集中校验”的目标是冲突的。

解决方式：
- 补充强类型枚举：
  - `ConditionType`
  - `ButtonListGroup`
  - `EventEffectType`
- `ItemTag` 新增 `Axe`
- `SkillDefinition.RequiredToolTag` 从 `string` 改成 `ItemTag`
- `EventConditionEntry.ConditionType` 从 `string` 改成 `ConditionType`
- `EventDefinition.ButtonListGroup` 从 `string` 改成 `ButtonListGroup`
- `ItemDefinition.ToolBonuses` 从字典改成 `ToolBonusDefinition`
- `NodeDefinitionBase.ExtensionData` 从 `Dictionary<string, object?>` 收紧为 `Dictionary<string, string>`
- 同步修正示例 YAML，使配置值与强类型枚举一致

主要修改文件：
- `Scripts/Core/Enums/ItemTag.cs`
- `Scripts/Core/Enums/ConditionType.cs`
- `Scripts/Core/Enums/ButtonListGroup.cs`
- `Scripts/Core/Enums/EventEffectType.cs`
- `Scripts/Core/Definitions/SkillDefinition.cs`
- `Scripts/Core/Definitions/EventDefinition.cs`
- `Scripts/Core/Definitions/ItemDefinition.cs`
- `Scripts/Core/Definitions/ToolBonusDefinition.cs`
- `Scripts/Core/Definitions/NodeDefinitionBase.cs`
- `Configs/Skills/skills.yaml`

- `Configs/Events/idle_events.yaml`
- `Configs/Items/items.yaml`
- `Configs/Zones/zones.yaml`


5. 已解决：金币与商店库存没有明确运行态的问题

问题说明：
原本金币和商店库存都停留在“设计上有这个概念”，但没有真正落到玩家运行时状态里。这样做商店、卖出、限量商品时，后面一定会返工。

解决方式：
- 新增 `PlayerEconomyState`，把金币从背包里拆出来
- 新增 `PlayerShopState`，记录每个 NPC 商店的剩余库存
- `PlayerProfile` 统一持有 `Economy` 和 `ShopStates`
- `ShopSystem` 现在已经可以做基础购买流程：
  - 检查 NPC 是否存在
  - 检查剩余库存
  - 扣金币或扣以物易物材料
  - 购买成功后发放物品
- `SellSystem` 现在已经可以做基础卖出流程：
  - 找出被标记为垃圾的物品
  - 按 `SellPrice` 结算
  - 把卖出收益加入金币钱包
- `PlayerShopState` 补了 `TryConsumeStock`

主要修改文件：
- `Scripts/Core/Runtime/PlayerEconomyState.cs`
- `Scripts/Core/Runtime/PlayerShopState.cs`
- `Scripts/Core/Runtime/PlayerProfile.cs`
- `Scripts/Core/Runtime/PlayerInventory.cs`
- `Scripts/Systems/ShopSystem.cs`
- `Scripts/Systems/SellSystem.cs`


6. 已解决：事件系统 effect 过窄，后续容易靠事件 ID 硬编码的问题

问题说明：
如果事件只能表达“扣物品、给物品、加经验”，以后做“加声望、解锁区域、完成任务、进入战斗”时，就会被迫写一堆 `if (eventId == xxx)`。

解决方式：
- 在 `EventDefinition` 里新增 `Effects`
- 新增 `EventEffectEntry`
- 事件效果类型现在支持：
  - 发放物品
  - 扣除物品
  - 发放金币
  - 扣除金币
  - 增加技能经验
  - 增加势力声望
  - 解锁区域
  - 标记事件完成
  - 标记任务完成
  - 预留开始战斗
- `ClickEventSystem` 改成统一解析这些 effect

主要修改文件：
- `Scripts/Core/Definitions/EventDefinition.cs`
- `Scripts/Core/Enums/EventEffectType.cs`
- `Scripts/Systems/ClickEventSystem.cs`
- `Scripts/Core/Runtime/PlayerProfile.cs`
- `Scripts/Systems/FactionSystem.cs`
- `Scripts/Systems/ZoneSystem.cs`


7. 已解决：排序职责混淆在静态定义层和玩家显示层的问题

问题说明：
`definition_order` 应该是静态定义顺位，通常只在配置层改；玩家在 UI 里临时拖动顺序，应该属于玩家态，而不该回写到定义注册表。

解决方式：
- 去掉 `ItemRegistry` 里“运行时修改定义顺位”的职责
- `PlayerItemState` 里改为 `PlayerDisplayOrder`
- `SortingHelper` 补上显示排序辅助方法：
  - `GetEffectiveDisplayOrder`
  - `SortNodesForDisplay`
- `ItemRegistry.GetChildren()` 明确只负责静态定义顺序
- `TreeValidator.ValidateDefinitionSets()` 在真正写入字典前先检查重复 ID

主要修改文件：
- `Scripts/Core/Runtime/PlayerItemState.cs`
- `Scripts/Core/Helpers/SortingHelper.cs`
- `Scripts/Core/Registry/ItemRegistry.cs`
- `Scripts/Core/Helpers/TreeValidator.cs`


8. 已解决：本轮复查时发现的几个直接 bug / 风险

8.1 `SaveManager.cs` 缺少 `PlayerProfile` 的命名空间引用

问题说明：
这会导致保存管理器在当前代码状态下有编译风险。

解决方式：
- 补上 `using Test00_0410.Core.Runtime;`

修改文件：
- `Scripts/Core/SaveLoad/SaveManager.cs`


8.2 一次性事件缺少系统层防重复保护

问题说明：
即使 UI 忘了隐藏按钮，只要从代码层再次调用，理论上还能重复触发一次性事件。

解决方式：
- 在 `ClickEventSystem` 新增 `IsAlreadyConsumed()`
- 对 `OneshotClick` 和 `RemoveAfterTriggered` 类型，在系统层二次拦截

修改文件：
- `Scripts/Systems/ClickEventSystem.cs`


8.3 技能经验变化后，没有同步刷新“可升级”状态

问题说明：
`PlayerSkillState.CanLevelUp` 有字段，但原本不会随着经验变化自动刷新，后续 UI 读这个字段时容易拿到过期值。

解决方式：
- 在 `SkillSystem.AddExp()` 和 `SkillSystem.TryLevelUp()` 后统一调用 `RefreshCanLevelUp()`

修改文件：
- `Scripts/Systems/SkillSystem.cs`


8.4 金币、库存、物品数量缺少基本边界保护

问题说明：
底层状态类如果不拦截负数或 0 这种非法输入，后面做 UI、调试按钮或脚本调用时会更容易出现脏数据。

解决方式：
- `PlayerEconomyState.AddGold()` 不再处理小于等于 0 的加钱
- `PlayerEconomyState.TrySpendGold()` 对 0 做安全处理，对负数做拦截
- `PlayerInventory.AddItem()` / `TryRemoveItem()` 增加非正数保护
- `PlayerShopState.SetRemainingStock()` 做最小值保护

修改文件：
- `Scripts/Core/Runtime/PlayerEconomyState.cs`
- `Scripts/Core/Runtime/PlayerInventory.cs`
- `Scripts/Core/Runtime/PlayerShopState.cs`

========================================
四、目前仍未完全实现，但已明确留出接口的部分
========================================

这几项不是这次 review 的遗漏，而是“当前项目本来就还处于骨架阶段”的内容：

1. `GameManager.LoadStaticData()` 还是占位
- 现在项目启动后，还没有真正把 `Configs/` 里的 YAML 读进注册表
- 也就是说：项目结构和数据契约已经整理好了，但“配置加载落地”还没接通

2. `YamlConfigLoader.LoadList<T>()` 还是占位
- 当前只会返回空列表
- 后续需要接入真正的 YAML 解析器（例如 YamlDotNet 或自定义解析）

3. 存档读写目前只有稳定接口，没有真实序列化
- `SaveManager.Save()` / `LoadOrCreateDefault()` 现在主要是保留契约
- 还没有真的把 `SaveData` 写入磁盘 JSON/YAML

4. 挂机收益结算逻辑还没真正实现
- `IdleSystem` 已经解决了“状态该放哪”“离线怎么补算”的架构问题
- 但“按技能等级、工具倍率、读条时间、背包上限实际结算原木与经验”还要下一步补

5. UI 和 SignalBus 还没有接通
- 现在大部分 UI 脚本、场景节点还是骨架
- `SignalBus` 已经建好，但系统还没有开始真正发信号、UI 也还没订阅

6. 本轮没有做真实编译验证
- 当前目录里没有 `.csproj`
- 所以这次我做的是“基于代码结构排查 + 语义修正”
- 真正要跑完整编译，还需要 Godot 重新生成 .NET 项目文件后再检查
