阶段一，我正在进行游戏开发的第一步，也就是最底层的数据结构的设计。

1. 物品通用类在实例化时需要为它指明所属子类；
例如：‘柴火’这一物品实例（Item），它属于‘木材’这一类节点（Category）下的子物品，而‘木材’这一类节点（Category）则属于‘采集素材’这一类子节点；

3. 物品通用类的实例化子类，在实例化（初始化）时需要提供哪些必要的参数，需要提供哪些可选参数？
实例物品的必要数据类参数，例如最关键的唯一标识符 (ID)，物品的可持有总上限（通常可以通过升级或扩展背包容量等方式逐步得到提升），玩家所持有的当前物品数量（不可超过总上限）。
实例物品的必要说明类参数，例如物品名称（通常是中文），物品的附加说明（通常是1-2句中文）。
特殊的bool型参数‘is_acquired’来记录‘玩家是否获取过该物品’，其默认值为false。该数据用于控制‘本物品是否应显示在玩家的道具列表中’。默认的false意味着‘当玩家第一次获取该物品之前，该物品在道具栏中不显示’。（如果获取过该物品，该物品将可以在物品字典列表界面供玩家查看其详细信息）。
特殊的bool型参数‘is_favorite’来记录‘玩家是否收藏了该物品’，其默认值为false。当玩家收藏该物品后，该物品将在物品栏被置顶显示（默认的显示顺序是物品顺位参数，被收藏的置顶物品则位于物品栏的最顶部）。
特殊的bool型参数‘is_abstract’来标记该物品是否允许直接持有，例如父类的物品分类节点（Category）的这项参数就应默认为false，而具体的物品（Item）则应默认为true，这些参数应该在初始化函数中自动设置。
除此之外，还需要一些可选的参数（这些参数在实例化物品（Item）时可以不用指定，但如果制定了它们的参数，这些物品将在游戏中带来额外用途），示例如下：
例如稀有度（每种物品在实例化时自带一个默认的初始稀有度，初始稀有度为最低级别，稀有度附带文字颜色、排序权重等附加属性）；例如物品类型标志（例如防具类物品的‘衣服’、‘头盔’，用于系统向玩家提供筛选查询）；例如物品价格（需要区分商店购买价格和玩家出售价格）；例如品质与耐久度（对于装备或工具）；例如物品所属的配方信息（用于合成系统，多物品锻造或制作某件成品）；例如物品获取来源（用于指示玩家该物品的获取渠道）；例如物品的详细信息（在字典界面提供可查看的详细描述文本）。
此外，这些从属于物品的参数还需要按照‘静态数据（ItemTemplate）’和‘动态实例（ItemStack）’进行区分。静态数据例如（唯一标识符ID、物品名称、物品价格、物品效果、物品文本说明）等，动态实例如（模版引用、当前数量、是否收藏、是否已获取过）。这样 is_acquired 实际上可以通过“玩家存档中是否存在该物品的 ItemStack（即使数量为0）”来判断，无需单独标志。
备注1：关于‘空物品分类’的处理——如果一个Category 下没有任何 Item 也没有子 Category，它是否应在 UI 中隐藏？这取决于它的is_visible或is_empty这两个bool字段其中之一是否为false。
备注2：物品上限可以通过扩展背包等方式扩容，因此物品上限并非是一个独立的参数，而是多个参数相加的总和，其中‘基础上限’依旧是从属于ItemDefinition的固定值，而‘扩容上限’则是属于玩家动态数据的内容。

4. 我想通过‘配置注册’而非‘手写实例’来完成每个新物品的初始化工作
简单说明：物品和分类不是在代码里直接创建，而是在配置里声明；程序启动时把这些声明统一读进来并注册成可查询的结构。
不是在代码里手写 `new Item(...)`、`new Category(...)` 来新增内容，而是把“物品定义”和“分类定义”写在配置里，程序启动时统一读取这些配置，然后自动注册进 `ItemRegistry`。通过这种方式，我在新增物品时改的仅仅是改的是“数据”，而不是“业务代码”。
1. 先准备‘定义数据’；
2. 游戏启动时读取配置文件（yaml），获取所有定义
3. 注册表自动建立分类关系、父子关系、索引、排序
4. 最后用‘物品唯一标识ID’来查物品
这样一来，我在新增‘馒头’时，只需要在配置文件中加一条新配置，不需要再写新的注册逻辑。
这样在我新增分类时，程序会自动执行‘新建一个CategoryDefinition资源’，填入‘唯一标识ID’，填如‘分类名称’，填入‘父分类ID’，填入排序；
新增物品时，程序会自动执行‘新建一个ItemDefinition资源’，填入‘唯一标识ID’，填入名称和文字说明，填入配置文件中手动指明的数据参数，自动填入配置文件中缺省的参数（默认参数），填入默认排序等由程序来控制的默认参数；
在游戏启动时：先‘注册表读取全部定义’，然后‘自动构建结构’，最后‘输出错误或打印树状物品结构清单’；

二：点击事件与挂机事件追加
我需要在上述‘角色属性系统’的基础上加入两种类型的玩家互动事件：一种是单次点击事件，一种是挂机循环事件。
游戏中所有的‘事件’均是以‘按钮’的方式出现在‘按钮列表’中。点击事件和挂机事件分别在两个不同的列表中。

一：单次点击事件。点击事件需要玩家用鼠标左键来点击‘单次点击事件按钮’。通过这类事件，玩家可以：
1. 一次性获取物品（例如点击鼠标左键砍树，点一下按钮获得一块木头）；
2. 提交某个道具来解锁新的事件（出现新按钮），这会消耗（扣除）玩家的道具（用于达成事件，例如合成道具，以物易物，赠送道具等事件）。如果玩家持有的道具不足，则该‘提交道具事件’的按钮处于‘无法点击’状态；通常在玩家满足条件并点击按钮后，该事件（按钮）将消失，并且不会再出现；
3. 开局需要提供一个‘一次性获取道具事件’（含按钮）：捡起‘石斧’事件。当玩家点击该按钮时，玩家获得道具‘石斧’，事件按钮小时，与此同时，玩家由于拥有了最低等级的斧头，因此具备了‘砍树’的条件。

二：循环挂机事件。该按钮在点击一次后，可以无限重复获取资源。
以‘砍树’为例，当玩家获得点击按钮的前提（通常是背包中获得某种道具，或是玩家的某项数值达标）后，‘挂机砍树’事件按钮将点亮，可供玩家选择挂机。
玩家点击挂机事件按钮后，将不需要再做其他操作。角色会自动开始进行砍树，在‘砍树技能等级’为1级时，每6秒完成一次读条结算，每次结算获得5个木头，并同时获得读条时间相等的技能经验值。技能经验值将自动在对应的‘技能经验槽’中累积，直到可以升级时，‘技能升级按钮’将‘亮起’可供玩家点击升级（若玩家不操作，不会自动升级）。
简单来说：系统需要准备‘获取石斧（一次性事件，点击后获得道具并让按钮小时）’、‘点击砍树（单次点击事件，按钮不消失）’、‘挂机砍树（点击后挂机按钮文字从‘挂机砍树’变成‘停止砍树’）’三个按钮。
‘点击砍树’属于单次操作，点一下获得一个木头，需要玩家不停地点。‘挂机砍树’属于挂机操作，按下后玩家可以不再管游戏内容，角色会自动进行‘挂机砍树’的循环，每次结算获得原木与技能经验值。当玩家包裹中的原木数量抵达物品上限后，挂机操作不会停止，只是无法再获取新的原木。

备注：原木的获取速度还可以进一步加强。初始获取的道具‘石斧’它会附带一个属性：‘原木获取率x100%’，也就是按基础获取量结算。当玩家后续升级为‘铁斧’、‘合金斧’后，其附带的属性将会使每次原木的获取率进一步提升（例如铁斧+200%，合金斧+400%）。
与此同时，后期也会有特殊道具可以增加砍树进度条（即结算读条时长）的效率，在无道具时，当砍树等级升到20级时，最快为2秒一次读条，但后期获得对应的效率加成道具后，这个读条时间也将进一步缩短，我需要你在设计‘读条效率’和‘原木获取率’时要考虑到来自原本技能之外的‘道具加成’的影响，为它们提供接口。


阶段三，游戏的具体玩法：
该挂机游戏是‘东方project’的同人游戏。主角是人间之里的村民，临危受命‘人间之里的外交官’，他的目的是与村子周围的各大妖怪势力打交道并获取其好感度——
好感度提升后，主角可以在不同势力的NPC处（按钮列表）购买不同道具、学习不同技能。当好感度达到100后，主角将作为外交官促使该妖怪势力与人间之里‘和平建交’，而该挂机游戏的主线任务就是让主角找到周围所有妖怪势力并让所有势力均与村子建交。
在主角寻找各大妖怪势力的过程中，会经历不同的‘危险区域’，也就是副本区域，在这些区域会经历战斗，通常需要通关10次（通关一次副本探索度+10%）后解锁新区域，因此在预留类时需要考虑以下类：
各大势力相关的类（如红魔馆、妖怪山、永远亭、命莲寺）及其所属的专属事件（专属一次性触发事件，与专属挂机类（打工）事件）以及对应的好感度系统（好感度系统可以用于解锁专属事件：如学习技能，交换道具，以及最终任务：和平建交）。
战斗系统相关的类：需要预留扩展接口，本次无需实现。
背包道具类：道具该如何在背包里呈现（前期所有道具均使用默认图片或无图片）。主角背包中的物品可以被标记为‘垃圾’后统一卖出，主角的物品售价为恒定价格，售卖后物品消失，不与其他卖方（NPC）产生联系。
买入交易系统：各大势力的不同NPC可能会提供不同的售卖物品，需要提供一套可选交易物品列表框架，该框架同样以‘按钮’的形式展出，鼠标悬浮在按钮上方才能查看道具详细信息。该货品信息所在按钮列默认出‘折叠隐藏’的状态，只有点击对应的商人NPC后，商人才会展开货品列供主角挑选，若主角做了其他动作（其他挂机事件）则商品自动重新折叠。注意：商人兜售的商品可能需要不同的货币甚至是道具，不能仅以货币作为买入条件，需要具备灵活定价（以物易物）的通用接口（默认是货币贸易，只有少数以物易物）。
事件系统：一次性按钮事件分为‘（满足前置动作才出现，满足条件才可以点击）点一下就消失（单次触发型事件）’和‘（如果满足条件即可）可多次点击（出售货物）’；挂机按钮则需要在开启挂机后在按钮上播放进度条来告诉玩家：正在循环执行该操作。注意：为了避免混淆，单次点击类普通事件（如买卖、合成）、一次性触发类事件（如交任务）、循环挂机事件（如砍树、打工）需要放在不同的按钮列表中，需要多组按钮列表可供复用。
LOG系统：该系统有两种作用，对开发者而言，在运行游戏期间的所有打印均在这里输出，LOG中输出各项操作的打印日志；在后期我会注释掉这些打印，该处文件信息仅用于向玩家展示说明挂机或单次操作中的日志信息（例如获取多少道具，完成什么任务等等）。

最后，我还需你帮我考虑UI方面的安排：前期仅测试游戏功能，所有UI均使用GODOT默认按钮+文字来显示，该游戏将作为一个无美术素材的纯文字+UI构成的挂机游戏。后期我再考虑将其替换为新的美术素材。
因此，在进考虑按钮与文字排布的情况下，我需要在多危险区域、多妖怪势力之间切换，每个危险区域和势力均有不同按钮事件。同时，我还需要一个始终显示在屏幕右侧的人物系统状态栏。以及‘背包’、‘属性’、‘战斗’‘任务’、‘字典’、‘成就’这些预留的空白标签页。
注意：战斗系统牵扯到具体的属性和数值设计，我需要你在架构中预留整个战斗系统的框架，确保其具备足够的可拓展性，战斗相关的类仅声明与备注不实现，确保我后续想要加入战斗系统时可以让你继续补充。
