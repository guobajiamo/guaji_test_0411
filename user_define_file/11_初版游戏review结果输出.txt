《初版游戏 review 结果输出》

review 范围说明：
- 本次重点审查“首版实例化后新增/重写的可运行部分”
- 也顺手检查了与之强相关的基础系统
- 本次审查以“静态代码 review + 工程结构核对”为主


================================
一、Findings（按严重度排序）
================================

结论先说：
本次静态 review 未发现“已经可以确定会导致首版无法启动”的硬错误。

但审查过程中发现了几处会影响运行规则正确性的逻辑问题，已直接在代码中修正：


1. 中：商店购买流程没有真正执行 NPC / 商品声望门槛检查
位置：
- `Scripts/Systems/ShopSystem.cs:26`
- `Scripts/Systems/ShopSystem.cs:88`

问题说明：
- `factions.yaml` 里已经给 NPC 和商品定义了 `required_reputation`
- 但旧实现只检查“NPC 是否存在”，不会检查：
  - NPC 是否真的开放商店
  - 玩家声望是否达到 NPC 门槛
  - 玩家声望是否达到商品门槛
  - 该商品是否确实属于这个 NPC

风险：
- 会出现“配置里明明写了声望限制，但运行时任何人都能买”的规则失效

处理结果：
- 已新增 `CanAccessNpcShop(...)`
- 购买前会检查：
  - NPC 是否存在且有商店
  - 商品是否属于这个 NPC
  - 玩家声望是否达到 NPC 门槛
  - 玩家声望是否达到商品门槛


2. 中：势力声望没有按上限裁切，也不会自动同步和平状态
位置：
- `Scripts/Systems/FactionSystem.cs:22`
- `Scripts/Systems/FactionSystem.cs:46`

问题说明：
- 旧实现里 `AddReputation(...)` 是直接加数值
- 没有参考 `FactionDefinition.MaxReputation`
- 也没有在达到 `PeaceThreshold` 时更新 `HasPeaceAgreement`
- `CanAccessNpc(...)` 也只检查 NPC 是否存在，不检查声望

风险：
- 会出现声望超上限、和平状态不变、NPC 解锁规则失效等问题

处理结果：
- 已加入声望上下限裁切
- 已在达到和平阈值时自动更新 `HasPeaceAgreement`
- `CanAccessNpc(...)` 已改为根据 NPC 所属势力和声望门槛判断


3. 低：旧版事件奖励模型没有真正使用掉落概率
位置：
- `Scripts/Systems/ClickEventSystem.cs:215`

问题说明：
- `EventRewardEntry` 里有 `DropChance`
- 但旧实现会无条件发放 `Rewards`

风险：
- 如果后面 YAML 里写了概率奖励，运行结果会与配置不一致

处理结果：
- 已补上 `DropChance` 判定
- 概率会先裁切到 `0.0 ~ 1.0`
- 小于 1 的奖励会走随机判定


4. 低：主界面布局资源缺失时没有显式回退提示，且“是否已搭建界面”的判断过于脆弱
位置：
- `Scripts/UI/MainUI.cs:91`
- `Scripts/UI/MainUI.cs:103`

问题说明：
- 如果布局资源路径填错，旧代码会直接用默认对象，但没有提示
- 同时旧代码用“`GetChildCount() > 0`”判断是否已经创建布局
- 这对以后在场景里加调试节点、装饰节点不够稳妥

风险：
- 出问题时不容易定位
- 后续自行调整场景时，可能误触发“认为布局已存在”

处理结果：
- 已在找不到布局资源时写入运行日志
- 已改为按 `OuterMargin` 节点是否存在来判断布局是否已创建
- 并对 `RefreshIntervalSeconds` 做了最小值保护


5. 中：项目当时缺失 Godot C# 工程文件，导致实际运行时无法加载脚本程序集
位置：
- 根目录缺少 `test_00_0410.csproj`

问题说明：
- 之前项目虽然已经有很多 `.cs` 文件，但没有真正的 Godot C# 工程文件
- 这会导致 Godot 无法编译出项目程序集
- 运行时表现出来的现象就是：
  - 自动加载脚本无法实例化
  - `MainUI.cs` 找不到对应类
  - 游戏停留在灰色界面

处理结果：
- 已新增 `test_00_0410.csproj`
- 并通过 `dotnet build` 成功生成程序集
- 随后再次用 Godot 启动项目，已确认主场景、自动加载和主 UI 可以正常拉起

================================
三、残余风险（本次未完全解决）
================================

下面这些不是“已经确认出错”，但仍然是当前版本需要记住的风险点：


1. 低：当前虽然已经完成 `dotnet build` 和 Godot 启动验证，但仍建议在编辑器里手动点一遍 UI 流程
原因：
- 我这边已经完成了：
  - `dotnet build`
  - Godot 无界面启动
  - 自动加载与主场景加载验证
- 但像“按钮点击后的交互体验”“标签页切换显示”“窗口模式下的视觉布局”这类内容，仍建议你本机再亲手点一次


2. 中：`Scripts/Core/Helpers/YamlConfigLoader.cs` 是自定义轻量解析器，不是完整 YAML 引擎
说明：
- 现在已经足够支持本项目现有 YAML 结构
- 但如果以后配置里出现更复杂语法，例如：
  - 更深层的复杂混合结构
  - 特殊转义
  - 多行文本块
  - 更复杂的锚点/引用
- 那就可能超出当前解析器的能力边界


3. 中：`Scripts/Core/SaveLoad/SaveManager.cs` 仍然是预留接口
说明：
- 现在 `SaveGame()` / `LoadGame()` 的流程和接口已经留好
- 但实际 JSON 序列化与反序列化还没有接
- 所以当前“手动保存 / 读档”更接近占位行为，不是正式成品功能


4. 低：`Scripts/UI/MainUI.cs` 目前采用定时整页刷新
说明：
- 这对首版小规模 UI 是可接受的
- 但如果后面按钮、面板、列表越来越多，刷新成本会逐渐上升
- 到时更推荐切成：
  - 事件驱动刷新
  - 只刷新受影响面板


================================
四、当前已经具备的排错日志点
================================

如果运行时出现问题，当前版本已经预留了几处比较重要的日志输出点：

1. 配置加载失败
位置：
- `Scripts/Autoload/GameManager.cs`

会输出：
- `GD.PushError("[GameManager] 配置加载失败：...")`
- 以及运行日志里的“配置加载失败：...”


2. YAML 核心配置缺失
位置：
- `Scripts/Autoload/GameManager.cs`

会输出：
- `警告：有核心配置未成功加载，请优先检查 YAML 格式和路径。`


3. 挂机条件失效时自动停止
位置：
- `Scripts/Systems/IdleSystem.cs`

会输出：
- `当前挂机条件已失效，系统已自动停止挂机。`


4. UI 布局资源找不到
位置：
- `Scripts/UI/MainUI.cs`

会输出：
- `未找到 UI 布局资源：...，已回退到脚本默认布局。`


5. 存读档当前仍是预留接口
位置：
- `Scripts/Core/SaveLoad/SaveManager.cs`

会输出：
- `[SaveManager] 预留保存接口，目标路径: ...`
- `[SaveManager] 预留读取接口，目标路径: ...`


6. 构建成功日志
位置：
- `dotnet build test_00_0410.csproj`
- `godot_run.log`

说明：
- 当前项目已能编译出：
  - `.godot/mono/temp/bin/Debug/test_00_0410.dll`
- 并且 Godot 启动日志里已经能看到：
  - 静态配置加载完成
  - 初版游戏已完成启动


================================
五、最终 review 结论
================================

当前这版“初版实例化程序”已经具备：

1. 真实配置加载
2. 默认 UI
3. 点击事件链路
4. 挂机结算链路
5. 技能升级链路
6. 基础背包 / 图鉴 / 势力 / 区域显示

并且本轮 review 已补上几处规则层面比较关键的问题。

所以当前结论是：

“从项目骨架阶段，已经推进到了首版可运行、可交互、可继续扩展的状态；但在你正式把它视为稳定版本前，仍建议用 Godot 编辑器做一次真实运行验证，重点确认 C# 编译、UI 刷新和 YAML 读取结果是否与本地编辑器环境一致。”
